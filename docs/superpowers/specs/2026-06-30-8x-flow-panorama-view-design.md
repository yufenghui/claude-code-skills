# 8X Flow 业务模型全景图 · 丰富化与防画坏 设计

- 日期：2026-06-30
- 范围：`packages/business-modeling` 插件（`bm-8x-flow` skill 及其共享 reference）
- 状态：待用户审查

## 一、背景与问题

实测 `bm-8x-flow` 产出的「最终业务模型全视图」存在两类问题（用户确认 A、B 都有）：

- **B（太简）**：全景图画得过于简单，丢了 8X Flow 骨架。
  - 根因：`reference/output-format.md`「报告结构层级」与 `skills/bm-8x-flow/SKILL.md §6` 都把该图定为「摘要全景层」——「只露核心履约项、不堆凭证字段/违约分叉」。Claude 照做，结果只剩「框 + 核心履约项 + 凭证串联线」，丢掉了履约请求→确认结构、凭证本身、变化点位置。
- **A（画坏）**：图没画完就结束，部分内容以纯文字流出 SVG。
  - 根因：`reference/svg-layout.md` 只讲「布局美观」（连线不穿节点等），**完全没有 SVG 语法合法性纪律**。Claude 手写 SVG 常踩：`<text>` 内嵌 HTML 标签（span/div/br 等）、坐标画出 viewBox、标签未闭合、`& < >` 未转义——任一条都导致 SVG 解析中断、下半部分矩形不渲染（留白/没画完）、`<text>` 文字流出成纯文本。（本次设计过程中，mockup 就因 `<text>` 内误用 `<span>` 复现了该 bug。）

## 二、目标

让 `bm-8x-flow` 实际产出时：

1. 全景图**完整表达 8X Flow 建模骨架**（不再是摘要级简化）——治 B。
2. 全景图（及所有手写 SVG）**不再出现解析中断/内容变文字**——治 A。
3. 全景图随业务结构（上下文数量）选择合适布局——给模板 + 变体。

并按用户要求**改名**：「最终业务模型全视图」→「业务模型全景图」。

## 三、设计

### 3.1 改名（全文统一）

「最终业务模型全视图」→「**业务模型全景图**」。该措辞也更贴合 `output-format.md` 已有的「全景图 / 摘要全景层」叫法。

涉及文件与位置见 §四。

### 3.2 补内容 / 治 B：全景图升级为「8X Flow 骨架全景」

把全景图从「摘要全景层（只露核心履约项）」升级为「8X Flow 骨架全景」。明确**必含元素清单**：

| # | 必含元素 | 说明 |
| --- | --- | --- |
| 1 | 合同上下文 + 两方角色 | 每个合同上下文作为一张卡片/分区，标两个角色 |
| 2 | 履约请求(时段) → 履约确认(时点) | 每个主要履约项画出请求→确认的异步结构 + 权利方/义务方（8X Flow 灵魂） |
| 3 | 凭证作为节点露出 | 履约请求=发起依据、履约确认=履约证明；不再只是一条串联线 |
| 4 | 标的物归领域上下文 | 标的物置于领域区，经凭证引用参与业务（业务不拥有其生命周期） |
| 5 | 变化点标注 | 角色化的履约确认（◆）+ 由哪些合同凭证扮演（反转依赖）；这是全景图独有、第二段单上下文看不到的跨上下文视角 |
| 6 | 四类边界分区 | 合同上下文=卡片框（服务边界）；履约上下文=卡片内分区（弹性边界）；领域上下文=独立区（弹性边界）；渠道上下文=独立区或标注 |

**仍留给第二段详细层**（保留全景 vs 详细的分工，不重复）：凭证关键字段（金额/时点具体值）、凭证追溯链、违约分叉完整路径。

> 与「详细局部层」的边界不变：全景画「结构骨架」，第二段画「字段/追溯/违约细节」。改的只是全景从「极简摘要」变为「骨架齐全」。

### 3.3 防画坏 / 治 A：SVG 产出纪律（所有手写 SVG 共用）

在 `reference/svg-layout.md` 新增「SVG 语法合法性纪律」节，并加入自检清单：

1. **`<text>` 内严禁嵌套 HTML 标签**（`span`/`div`/`br`/`b`/`i`/`em`/`strong` 等）。文字样式只用 SVG 属性（`font-size`/`font-weight`/`fill`）或 `<tspan>`；分段另起一个 `<text>`。
2. **特殊字符转义**：`&`→`&amp;`、`<`→`&lt;`、`>`→`&gt;`。
3. **所有坐标必须在 viewBox 内**；内容多时**增大 viewBox 高度（或宽度）**，绝不压缩字号到不可读或裁切内容。
4. **标签闭合、属性引号成对**；每张 `<svg>` 顶部放 `defs`（阴影 + 箭头 marker）。
5. **产出后自检渲染完整性**：图是否完整渲染、有无文字流出 SVG、有无裸露文本节点。发现中断立即定位语法错误并修。

`reference/html-report.md §七 检查清单` 同步加这几条。

### 3.4 画法落地：标准布局模板 + 按上下文数量变体（用户选定方案）

在 `reference/html-report.md` 新增「**全景图画法**」节，给可填充的 SVG 模板（节点片段复用 §四片段库），Claude 按业务内容填充。模板本身是合法 SVG，从根本上降低画坏概率，同时内置骨架保证丰富度。

**两个布局变体 + 选择判别**：

| 业务形态 | 布局变体 | 结构 |
| --- | --- | --- |
| 单合同上下文（1 个） | **纵向型** | 单个合同上下文纵向展开（履约项纵向排列：请求→确认）+ 领域区（标的物）+ 变化点标注（若有） |
| 多合同上下文（≥2） | **网格型** | 顶部领域区 + 合同上下文卡片网格（2–3 个用单行/2×2，4+ 用多行网格；每卡含履约请求→确认 + 凭证 + 权责方）+ 跨卡片变化点连线 + 底部图例 |

- 网格型模板**以已验证过的外卖平台示例图为准**（4 上下文 2×2，渲染正确、合法 SVG，存于本次 brainstorm 的 `.superpowers/brainstorm/.../panorama-rich-v2.html`，实现时整理进 `html-report.md`）。
- 平台枢纽型（多方、枢纽方在多份结算合同作对手方）：用网格型，对手方结算合同用粗线、核心合同用粗边框突出。
- 必含元素清单（§3.2）对两个变体都适用。

## 六、验证

1. **改名一致性**：全仓库 grep「最终业务模型全视图」应无残留；新名「业务模型全景图」在各文件一致。
2. **治 B**：按新 spec 跑一个示例（如外卖平台），全景图含 §3.2 全部必含元素，不再是「框 + 核心履约项」。
3. **治 A**：产出的全景图 SVG 通过合法性自检（无 HTML 标签混入 `<text>`、坐标在 viewBox 内、特殊字符转义），浏览器完整渲染、无文字流出。
4. **变体**：单上下文业务产出纵向型、多上下文业务产出网格型，判别正确。
5. **本地测试**：`claude --plugin-dir ./packages/business-modeling` + `/reload-plugins`，跑一遍确认。

## 七、风险与对策

- **模板僵化**：非常规业务结构（极多上下文、嵌套平台）可能不适配标准网格 → 对策：模板给出「内容多则增大 viewBox / 多行网格」的弹性指引，且必含元素清单保证内容齐全，布局可调。
- **skill 篇幅增长**：新增模板 + 纪律使 `html-report.md`/`svg-layout.md` 变长 → 对策：模板用可复用片段库引用，纪律条目精简（5 条），控制增量。
- **CLAUDE.md 约定**：项目要求手写 SVG、CSS counter 标号、双轨产出——本设计遵守，无冲突。
