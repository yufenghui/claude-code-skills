## 课程脉络

课程以“旧约 / 新约”划分，对应**前云时代**与**云时代**两套架构约束下的业务建模方法。主线是：先讲清楚 DDD 这套“定义问题”的方法，再讲云时代架构约束（弹性优先）如何改变建模，最后给出徐昊自创的 8X Flow、中台建模、魔球建模法等可落地方法。

### 旧约：前云时代的领域驱动设计（单体 / 多层架构）

围绕 Eric Evans 的 DDD，核心是**知识消化（Knowledge Crunching）**，可总结为“**两关联一循环**”：

- **两关联**：模型 ↔ 软件实现关联；统一语言 ↔ 模型关联。
- **一循环**：提炼知识的重构循环（通过统一语言讨论需求 → 发现模型缺失 → 精炼模型 → 改变统一语言 → 再验证）。

关键概念与方法：

- 统一语言（Ubiquitous Language）：业务方与研发达成共识的桥梁，比模型更易传播。
- 富含知识的模型（充血模型）vs 贫血模型；实体 / 值对象 / 聚合 / 聚合根。
- 跨越现实的障碍：性能 vs 模型、富含知识 vs 代码坏味道、架构分层是否够用。
- 事件流的获取方法（事件风暴等）。
- 将模型落地为 RESTful API。

### 新约：云时代的业务建模（微服务 / 云原生架构）

云时代引入新架构约束——**弹性优先原则**，建模方法随之演化：

- **弹性边界 vs 业务边界**：用弹性划分系统，区分业务系统与领域系统。
- **8X Flow**（徐昊发明）：以**权责追溯**为核心，提取业务模式；通过引入**变化点**构建可复用的业务模型，天然适配业务平台 / 中台，并易映射到微服务。
- **中台建模**：厘清中台本质，寻找可复用的业务模式。
- **模型落地为微服务**。
- **SaaS 化与魔球建模法（Money Ball Offering Modeling）**：从运营角度构造 SaaS 化服务。
