# 四色建模法引导

四色建模法是徐昊自创的**强分析法**：从业务底层逻辑（财务权责）出发**推演**领域模型，而非头脑风暴式发散-收敛。产出可验证、易被业务方接受为统一语言的模型。

## 何时使用

适用：
- 对**业务系统**（与盈利相关）做领域建模
- 需要从收入流 / 成本结构发现关键领域事件与凭证链
- 事件风暴结果太随意、过度依赖主持人经验

不适用 / 改用他法（判断依据见 `../../reference/methods.md` 的方法选择指南）：
- **领域系统**（运营无关，如搜索引擎 / CMS）→ 用对象建模 / 知识消化
- 建模**盈利流程 / 合同履约 / 变化点** → 用 `bm-8x-flow`（四色法常作为其中"发现凭证"的子方法）
- 需要与业务方快速发散共创 → 事件风暴

## 前置：先抓业务模式

业务模式（通过收入流与成本结构体现）在企业生命周期内**相对稳定**——从稳定部分建模。先确认：
- 这项业务有哪些**现金往来**（收入 / 支出）？→ 对应**收入流**
- 哪些环节靠 **KPI / 目标-实际对比**驱动（无直接现金往来）？→ 对应**成本结构**

## 三条核心逻辑（发现凭证的依据）

1. **现金收入 → 承担义务**：拿钱办事，需收集证据证明义务履约成功。
2. **现金支出 → 拥有权利**：花钱消灾，需检查对方是否按时履约。
3. **无现金往来 → 目标-实际对比（KPI）**：设目标追踪实际，产生类似履约的约束。

前两条源自现金日记账（收入流），第三条源自成本结构。

## 建模工作流

复制此 checklist 逐项推进，每步与业务方确认后再进入下一步：

```
四色建模进度：
- [ ] 1. 识别现金往来 / KPI 入口
- [ ] 2. 构造凭证，列关键数据项
- [ ] 3. 追溯前序凭证（计算逻辑 + 时标顺序逻辑）
- [ ] 4. 找履约凭证，约束时序
- [ ] 5. 形成业务脊梁
- [ ] 6. 角色扮演验证（重点跑异常追溯）
- [ ] 7. 补全角色 / 参与者 / 标的物 / 描述对象，归入四原型
```

**步骤 1：识别入口**　列出所有现金往来（如读者购专栏、作者分成）与关键 KPI，每个都是一条凭证链的起点。KPI 入口按"目标-实际对比"建模（目标凭证 + 实际履约凭证 + 偏差追溯，结构见 `../../reference/concepts.md` 的"KPI 凭证链结构"）。

**步骤 2：构造凭证，列关键数据项**　为每个入口构造凭证，明确**关键数据项**（通常是**时间点 + 金额**）。

**步骤 3：追溯前序凭证**　对每个关键数据项问"它从哪来"，追溯直到源头。来源有三类（详见 `../../reference/concepts.md` 的"关键数据项的三类来源"）：① 前序凭证的数值；② 人工输入；③ **由前序凭证中的规则 / 算法计算得到**（如骑手配送费 = f(距离, 时段, 天气)，追溯到计费规则凭证）。必须记录两类逻辑：
- **计算逻辑**：如 `amount = sum(payment.amount) * percentage where payment.paid_at in (last_paid, payment_days)`
- **时标顺序逻辑**：如 `Order.placed_at > ColumnQuote.available_at`（定价 14:00 变更，13:59 下单须按旧价 199 元）

**步骤 4：找履约凭证**　对每条现金权责，问"付钱得到了什么权利 / 收钱承担了什么义务"，权利或义务对应一张履约凭证（发现规则与双侧范例见 `../../reference/concepts.md` 的"履约凭证的发现规则"），并**约束时序**（如订阅 `started_at` 必须晚于支付完成，差一毫秒也不行）。

**步骤 5：形成业务脊梁**　由关键数据项彼此关联的凭证链即**业务脊梁**，等价于领域事件流，捕捉业务收入流全过程。

**步骤 6：角色扮演验证（勿跳过）**　先为异常 / 逆向流程建模（退款、取消等也是凭证，范例见 `../../reference/concepts.md` 的"异常 / 逆向凭证"），再把凭证转成真实单据，让业务方**模拟经营**，重点跑**异常追溯**（定价变更 / 分成争议 / 退款 / 超时等），验证业务脊梁能否证明权责、经受财务追溯考验。

**步骤 7：补全模型，归入四原型**　按关键数据项关联把模型连起来，依次补：角色 → 参与者 → 标的物 → 描述对象。每个元素归入四原型之一（详见 `../../reference/concepts.md` 的"四原型"，含描述型 vs 时标型判别）：
- **时标型（红 / 粉）**：凭证，独立参与权责追溯，建模的锚点
- **描述型（蓝）**：仅被引用的目录 / 规则模板（如报价单、优惠券），承载共享属性
- **角色型（黄）**：参与方在特定凭证中扮演的角色
- **参与者型（Party-Place-Thing，绿）**：人 / 组织 / 地点 / **物**——**标的物（商品、餐品等）归此类**

## 两条硬约束（违反则模型无效）

1. **凭证必须围绕现金往来或 KPI**——不能凭空造凭证。
2. **凭证间必须通过关键数据项明确关联**——含计算逻辑与时标顺序逻辑。

## 产出（双轨）

按 `../../reference/output-format.md` 的产出规范交付**双轨**产出（必含表格 + 图 + 端到端走查 + 判别说明）：

- **终端轨**（Markdown，对话 / 终端直接渲染）：图表用**纯 ASCII**（画法见 `../../reference/terminal-diagrams.md`），**不用 Mermaid**。
- **报告轨**（HTML 文件，交付业务方）：图表用**手写 SVG**（画法见 `../../reference/html-report.md`，套用 `../../reference/templates/report.html`）。

两轨语义一致（同样的凭证、线型、时标约束），内容都含：

1. **业务脊梁（凭证链）**：凭证表（关键数据项 + 计算逻辑 + 时标顺序逻辑）+ 凭证链图（实线 = 数值 / 时标来源，虚线 = 规则来源）。
2. **异常 / 逆向凭证**：退款、取消等（范例见 `../../reference/concepts.md` 的"异常 / 逆向凭证"）。
3. **KPI 凭证链**（若场景含 KPI 入口）：目标凭证 → 实际履约凭证 → 偏差追溯。
4. **四原型分类表**：元素归入时标型 / 描述型 / 角色型 / 参与者型（**标的物归参与者型 Party-Place-Thing**）。
5. **端到端走查**：正常流程 + 至少 1 个异常场景，验证业务脊梁自洽。

可作为统一语言提案交业务方确认。完整双轨样例见 `../../examples/B2C商城-建模方案/`（`方案.md` 终端 ASCII + `报告.html` 手写 SVG）；旧 Mermaid 样例见 `../../examples/外卖平台-建模实测方案.md`。
