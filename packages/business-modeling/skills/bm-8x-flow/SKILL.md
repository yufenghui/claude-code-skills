# 8X Flow 引导

本 skill 引导你为**业务系统**（与盈利相关的系统）建模：以**合同**为最小业务上下文、**权责履约**为最小业务交互，产出合同上下文图、履约项、违约流转、变化点与弹性边界，用于划清服务边界、预判变化点。其视角源自财务 / 合同审计——不了解领域，也能凭权责与合同系统理解业务逻辑；核心是**合同履约**（凭证追溯由四色建模承担，见 `bm-four-color`）。

## 运行方式：增量引导 + 实时可视化

按 `../../reference/live-session.md` 运行——参照 superpowers brainstorming：
- **just-in-time 启动 live server**（需要看图时才提议，不默认开），浏览器实时显示报告。
- **一次一问**，按章节序列（见 live-session §4）增量推进，每章产出后浏览器自动刷新、用户确认再继续。
- 每章产出：**详细只写 HTML**（图/表/判别写进 report.html，浏览器实时显示），**终端只给一两句摘要**（做了什么 + 关键判别 + 要确认的问题），不重复完整方案，省 token。

8X Flow 的五步法对应章节序列的：第 4 章替换为"合同上下文图 + 主要履约项表 + 凭证追溯图"、第 5 章替换为"违约流转图"，并加"变化点图"、"弹性边界 / 服务图"与"最终业务模型全视图"（收尾整合）。

## 何时使用

适用：
- 建模**业务系统**（与盈利相关）
- 设计微服务 / 服务边界（合同上下文天然映射服务边界）
- 构造业务平台 / 中台（需识别可复用模式与变化点；**中台建模见末尾「中台化」延伸章节**，不在主体五步法展开）
- 需从业务结构预判变化点与弹性边界

不适用 / 改用（判断依据见 `../../reference/methods.md`）：
- **领域系统**（运营无关，如搜索引擎 / CMS）→ 对象建模 / 知识消化 / 四色
- 只需找领域事件 / 凭证链（不涉及合同履约）→ `bm-four-color`

## 前置：先判别领域系统 vs 业务系统

- 换**运营方式**（如加企业版）会变的功能 = 业务逻辑 → 用 8X Flow
- 换**内容 / 载体**（红薯→土豆、网文→网剧）仍成立的赚钱逻辑 = 业务逻辑
- 运营无关、描述某个问题域的 = 领域逻辑 → 用对象建模

不确定时默认：与盈利相关就用 8X Flow。

## 元模型（核心概念，详见 `../../reference/concepts.md`）

- **合同上下文**：最小业务上下文，只在两方之间（多方 = 多份双方合同），含**两个角色**
- **履约请求**：一个**时间段**，由权利方发起，要求义务方在规定时限内完成
- **履约确认**：一个**时间点**
- **凭证**：履约请求 = 发起依据，履约确认 = 履约证明；在权责范围内**按四色建模寻找**
- **标的物**：凭证涉及的对象（如商品），来自领域系统

Request-Confirmation 是**异步结构**——这是业务的真实状态，不是技术选择。履约失败触发新的履约项，而非自动修正。

## 五步建模工作流（增量推进，每步对应 report.html 章节）

启动 live server（见 `../../reference/live-session.md`）后，按以下五步逐章增量产出——每步问关键问题 → 更新 report.html → 浏览器自动刷新 → 用户确认 → 下一步：

```
8X Flow 进度（每步产出一章，增量追加进 report.html）：
- [ ] 1. 找合同上下文，明确参与方 → 合同上下文图
- [ ] 2. 找主要履约项，按四色找凭证 → 主要履约项表 + 凭证追溯图
- [ ] 3. 找违约情况，设新履约项，按四色找凭证 → 违约流转图
- [ ] 4. 重复步骤 3，直到不得不打官司为止 → 违约流转图（补全）
- [ ] 5. 把参与方与标的物划入领域边界 → 弹性边界 / 服务图
- [ ] 收尾：整合所有合同上下文 + 凭证串联 + 领域系统 → 最终业务模型全视图
```

**步骤 1：找合同上下文，明确参与方**　合同只在两方之间；涉及多方时拆成多份双方合同。**平台 / 枢纽型业务**（外卖、电商、打车）判断枢纽方出现在哪些合同：识别每个盈利流的实际收款方 / 付款方——他们就是结算合同的对手方；只撮合不直接收付的是支撑方，不进核心合同（见 `../../reference/concepts.md` 的"枢纽参与方 / 平台型业务"）。

**步骤 2：找主要履约项，按四色找凭证**　先按"围绕该合同核心对价 / 收入流的履约项即主要履约项"筛选（见 `../../reference/concepts.md` 的"主要履约项 vs 违约履约项"）。对每项权责明确权利方 / 义务方；履约请求 = 时间段，履约确认 = 时间点。**找凭证的最小子步骤**（四色法核心）：① 围绕该履约项找现金 / KPI 凭证；② 列关键数据项（时间点 + 金额）；③ 追溯每个关键数据项的来源（前序凭证数值 / 人工输入 / 规则计算，见 `../../reference/concepts.md` 的"关键数据项的三类来源"），**产出凭证追溯图**（凭证链图 + 凭证表，复用 `bm-four-color` 画法：实线 = 数值 / 时标来源，虚线 = 规则来源）。追溯范围限定**主要履约项的核心凭证链**；违约履约项凭证在步骤 3 的违约流转图节点上标注，不入此图。若需完整全量凭证链，调用 `bm-four-color`。

**步骤 3：找违约情况，设新履约项**　对每个主要履约项寻找违约场景。判别处理方式（见 `../../reference/concepts.md` 的"独立履约项 vs 自动作废"）：违约需对方**主动作为**（退钱、赔付）→ 设独立履约项，可继续递归；违约只是**状态翻转**（如订单关闭）且无需对方确认 → 自动作废，到此终止。违约责任也需合同预先覆盖。按四色找凭证，**违约履约项的凭证在违约流转图节点上标注**（不单独追溯，见步骤 2 边界）。

**步骤 4：重复步骤 3，直到不得不打官司**　违约的承担继续违约 → 再触发新履约项 → 最终诉诸法律（终极履约）即为**终止条件**。

**步骤 5：划领域边界**　把参与方与标的物划入领域边界（标的物常来自领域系统，通过凭证引用参与业务）。

> 示例（专栏订阅合同，参与方：读者 ↔ 极客时间）：主要履约项 = 支付订阅费（权利方极客时间）+ 访问付费内容（权利方读者）。违约：未限时支付 → 合同自动作废（无需独立权责项）；断更 → 退钱 + 重上架免费，若拒不退钱 → 只能打官司（终止）。

## 引入变化点（8X Flow 的核心价值）

跨合同上下文的凭证引用 = 上下文间的**依赖关系**。**核心合同依赖支撑合同**（如专栏订阅依赖各种支付合同）是坏味道——让核心逻辑依赖支撑逻辑。

解法：**反转依赖——让履约确认角色化**（能力供应商模式），让其他合同中的凭证来扮演这个履约确认，从而在核心合同中引入**变化点**。

> 示例：把"专栏付款确认"角色化，由移动支付合同 / 预充值合同等不同合同的凭证来扮演 → 一处变化点支持多种支付方式，核心合同不再逐个依赖新支付合同。

## 弹性边界落点

- 合同上下文 **≠** 弹性边界（同一合同的不同履约项弹性诉求不同，如访问内容比支付容量诉求大）
- **履约上下文 = 弹性边界**
- 领域逻辑边界也是弹性边界

依此切分领域与业务边界，即可应对云时代建模的四挑战（反映弹性边界、切分后维护一致性、异步解读业务、异步异常维护一致性）。**映射到服务的启发式**：按履约项的容量 / 频次聚类成服务（如高频大容量的"配送履约"独立，低频批处理的"结算履约"独立），而非按合同整体切。

## 补充：合同前与无合同场景（第 16 讲）

- **渠道上下文（合同前）**：邀请投标（RFP）→ 投标（Proposal）→ 签约 → 履约。渠道上下文是系统**第二大变化点**，弹性诉求与履约差距大，必须与履约上下文分离。
- **无对外合同的四种情况**：① 内部绩效系统（用绩效协议 + 进度检查建模，权责 = KPI/OKR）；② 纯领域系统（按领域建模）；③ 工具（胶水代码）；④ 你没意识到其中有合同（重新建模，把隐藏合同找出来）。

## 产出（HTML 报告为主，终端摘要）

按 `../../reference/output-format.md`（含**视角纪律：业务为主角，方法为隐形工具**）+ `../../reference/live-session.md`：**HTML 报告含完整方案**（图/表/判别，live server 实时预览），**终端每章只给一两句摘要**（不重复完整方案，省 token）。内容项（写在 HTML 里）都含：

1. **合同上下文图**：参与方 + 双方合同，**标出平台 / 枢纽方的对手方结算合同**（粗线）与仅撮合的合同区分。
2. **主要履约项表**：每项权利方 / 义务方 + 履约请求（时间段）/ 履约确认（时间点）+ 凭证。
3. **凭证追溯图**：主要履约项的核心凭证链——追溯每个关键数据项来源（前序凭证数值 / 人工输入 / 规则计算），凭证链图（实线 = 数值 / 时标来源，虚线 = 规则来源）+ 凭证表（计算逻辑 / 时标顺序逻辑）。复用 `bm-four-color` 画法；完整全量凭证链调 `bm-four-color`。
4. **违约流转图**：每个违约 → 独立履约项 vs 自动作废的判别路径，终止于打官司；违约履约项凭证在节点上标注。
5. **变化点图**：角色化履约确认图（哪些履约确认被角色化、由哪些合同凭证扮演）为主 + 关键变化点表（变化点 / 角色化的履约确认 / 扮演凭证来源合同 / 业务意义）为辅。
6. **弹性边界 / 服务图**：按履约上下文划分弹性边界 → 服务映射（区分业务系统 / 领域系统）。
7. **最终业务模型全视图**：整合所有合同上下文 + 凭证跨上下文串联 + 领域系统（标的物归领域系统，经凭证引用参与业务），呈现业务如何将领域系统转化为可盈利产品。单合同场景即该合同完整模型；多合同场景为多上下文整合全景。落实 `../../reference/output-format.md` "业务模型图（全景图）"要求。
8. **端到端走查**：正常流程 + 至少 1 个违约场景，验证模型自洽。
9. **业务组件设计**（可选延伸）：领域组件清单、业务组件清单、组件划分总图、组件调用时序图——见下文「业务组件设计」。
10. **中台化：宏流程提取**（可选延伸）：宏流程图 + 中台业务模型——仅当目标是下沉中台 / 多前台复用时产出，见下文「中台化」。

**章节结构是默认推荐、可按业务增删/重排**（标号 CSS counter 自动）；看完整效果用 live server 实时生成（见 `../../reference/live-session.md`）。

## 业务组件设计（五步法的延伸产出）

五步法产出合同上下文 / 主要履约项 / 违约流转 / 变化点 / 弹性边界后，可进一步把系统划分为「领域组件」与「业务组件」，作为后续技术设计的结构骨架。本节为**可选延伸**——当需要从业务模型过渡到设计 / 实现时产出。

### 两类组件（呼应「领域逻辑 vs 业务逻辑」）

- **领域组件**：运营无关的领域逻辑 / 模型——标的物、领域状态机、主数据。**换运营方式也不变**。
- **业务组件**：运营特定的履约逻辑——履约流程、编排。**随运营而变**。
- **关键关系**：业务组件通过**引用领域组件的标的物**参与业务；**业务组件编排领域组件**（领域组件提供“是什么”，业务组件编排“怎么做”）。

### 划分步骤

1. **识别领域组件**：从凭证的标的物、领域状态、主数据中，提取运营无关的领域模型（聚合根、状态机、目录）。
2. **识别业务组件**：从主要履约项 / 履约流程，提取业务能力单元。
3. **明确编排关系**：每个业务组件编排哪些领域组件、与哪些业务组件协作。
4. **环节独立性判别**（见下）：不独立的环节融入相关业务组件。

### 关键判别：环节独立性（业务语义连续性约束）

并非所有环节都能独立成组件。判别测试：

> **若某环节独立成组件后，会丢失其业务语义（无法确定业务含义 / 方向），则它必须融入相关业务组件，不能独立。**

典型：具有「方向性」或「上下文绑定」的环节。例如追溯场景的「扫码采集」——入库采集与出库采集的码状态翻转方向相反，若把采集抽成独立共享组件，“码被采集”将无从判断是入库还是出库，方向语义丢失。故采集必须融入出入库追溯，不独立成组件。

> 这是「业务语义优先于物理复用」：两个环节物理相似但业务语义不同，应分属各自业务组件，而非为复用抽象成共享组件。

### 产出（写入报告「业务组件设计」章节）

1. **领域组件清单**（表）：组件 / 领域职责 / 核心领域模型（聚合根、状态机）/ 关键状态与规则。
2. **业务组件清单**（表）：组件 / 业务职责（履约流程）/ 编排的领域组件 / 协作的业务组件。
3. **组件协作图**：
   - **组件划分总图**（静态结构）：业务组件编排领域组件，标注标的物引用与基准引用。
   - **组件调用时序图**（动态交互）：以编排者（业务组件）为生命线发起调用，体现调用方向与参与时段。**时序图优于横向流程图**——“协作”本质是交互 / 调用，时序图能表达编排关系与调用方向，而横向流程图把协作扁平化为步骤序列、丢失编排层级。
4. **组件边界如何帮助设计**：领域组件 → 领域模型设计（聚合根、状态机、领域规则）；业务组件 → 履约流程设计（步骤、编排、权责）；**一个组件 = 一个设计 / 实现单元**。

### 与五步法的关系

- 弹性边界 ≈ 组件边界候选
- 标的物（来自领域系统）→ 领域组件
- 主要履约项 / 履约流程 → 业务组件
- 变化点 → 可在组件边界引入（业务组件定义端口契约，适配器实现）——但这是技术设计，不在本节展开

### 原则

- **去技术化**：业务组件设计是业务 / 领域设计，**不含技术实现**（端口 / 接口 / 部署 / 事件机制等留给后续技术设计）。它是技术设计的**输入**，不是技术设计本身。
- **业务语义优先**：组件划分服从业务语义（如方向性约束），不为物理复用破坏语义。
- 组件图遵循 `../../reference/svg-layout.md` 布局原则（分区 / 流向 / 走通道 / 先线后点）。

### 示例（追溯码系统）

- 领域组件：追溯码领域（码状态机：入库核注 / 出库核销、包装层级）、主数据领域（货主 / 商品 / 参与方）。
- 业务组件：入库追溯（含入库采集 + 收货确认）、出库追溯（含出库采集 + 复核确认）、监管合规上报、质量异常管控。
- 环节独立性：采集因方向性约束，融入入库 / 出库追溯，不独立。

## 中台化：宏流程提取（五步法的延伸产出）

当五步法已产出完整业务模型，且目标是把可复用业务模式**下沉为中台、支撑多个前台团队**时，可进一步从业务模型中**提取宏流程**，作为中台的业务模型。本节为**可选延伸**——仅当上述目标成立时产出；常规业务建模到此不必进入。

> 前置：中台是 8X Flow 的下游应用（17 / 18 讲开篇："如何使用 8X Flow 建模中台系统"）。进入本节前，先理解中台概念（中台 / 前台 = 团队 / 效能-自主 / 宏流程，见 `../../reference/concepts.md` §六）。

### 中台与宏流程（先建立概念）

- **中台**：支撑多个**小而灵活的前台团队**的软件平台。注意"前台"指**团队**（按城市 / 品类划分的独立运营单元），不是软件。
- **宏流程 = 特定场景中的业务模式**：从多个类似业务中提炼的共有底层"核心业务模式"。业务上 = 宏观流程；技术上 = 业务流程的**模板**，靠**配置点 + 扩展点**在不同前台实例化。
- **中台 ≈ BaaS**（产生 SaaS 的 SaaS）：只有后端部分，不新增前端入口。
- 详见 `../../reference/concepts.md` §六。

### 提取步骤（18 讲）

1. **从已产出的 8X Flow 业务模型出发**——宏流程从**已存在模型提取**，不直接建模（直接建模易过度抽象）。
2. **确定要复用的业务模式**：哪个合同上下文承载了值得跨前台复用的核心模式。
3. **定位对应合同上下文**：合同流程是业务流程的模板（合同流程是因、业务流程是果），是宏流程的最佳候选。
4. **将履约确认项角色化**，引入变化点（复用五步法的"履约确认角色化"手法）。
5. **在不同前台场景扩展 / 替换变化点**：同一合同流程 + 不同领域系统扮演 → 实例化成不同业务流程（如出行合同 + 网约车 / 出租车 / 共享单车领域系统）。
6. **各上下文继续 8X Flow 建模**，直到能解释清楚不同前台如何扩展宏流程。

### 关键判别与约束

- **只对履约确认项角色化，不得对履约请求项角色化**——后者会破坏权责语义（18 讲思考题）。
- **防过度抽象**：宏流程要在"可跨场景复用"与"对具体业务决策仍有指导意义"之间取平衡；从已存在模型提取是抗过度抽象的主要手段。
- **业务模式 vs 领域功能**：希望复用业务模式 → 用合同上下文建模（业务系统）；只关心功能 → 当领域系统处理（对象建模）。两者可分离。

### 与业务组件设计的区别

- **业务组件设计**：把**一个**业务系统拆成领域 / 业务组件（单系统内部结构）。
- **中台化**：把**多个**业务的共性模式提取成可复用宏流程（跨业务 / 跨团队复用）。
- 两者并列，按目标选用：内部结构 → 业务组件设计；跨前台复用 → 中台化。

### 产出（写入报告「中台化」章节，仅当启用时）

1. **宏流程图**：提取出的合同上下文 + 角色化变化点 + 多个前台场景的实例化（同一宏流程展开成不同业务流程）。
2. **中台业务模型**：宏流程 + 各变化点的扩展 / 配置方式。
3. **判定依据**：为何选中该合同上下文作为宏流程、哪些履约确认项被角色化、为何不能角色化履约请求项。

### 原则

- **后置**：先有完整业务模型，再提取宏流程；不在五步法中途扯中台。
- **去技术化**：与业务组件设计一样，本节是业务 / 领域设计，不含技术实现（配置点 / 扩展点的技术落地留给后续技术设计）。
