# 8X Flow 引导

8X Flow 是徐昊自创的业务建模法，借鉴**财务审计 / 合同审计 / 职业经理人**看企业的视角：不了解领域，也能凭**权责与合同**系统理解业务逻辑。其核心是**合同履约**（凭证追溯则由四色建模承担）。

## 运行方式：增量引导 + 实时可视化

按 `../../reference/live-session.md` 运行——参照 superpowers brainstorming：
- **just-in-time 启动 live server**（需要看图时才提议，不默认开），浏览器实时显示报告。
- **一次一问**，按章节序列（见 live-session §4）增量推进，每章产出后浏览器自动刷新、用户确认再继续。
- 每章产出：**详细只写 HTML**（图/表/判别写进 report.html，浏览器实时显示），**终端只给一两句摘要**（做了什么 + 关键判别 + 要确认的问题），不重复完整方案，省 token。

8X Flow 的五步法对应章节序列的：第 4 章替换为"合同上下文图 + 主要履约项表"、第 5 章替换为"违约流转图"，并加"弹性边界 / 服务图"与"变化点清单"。

## 何时使用

适用：
- 建模**业务系统**（与盈利相关）
- 设计微服务 / 服务边界（合同上下文天然映射服务边界）
- 构造业务平台 / 中台（需识别可复用模式与变化点）
- 需从业务结构预判变化点与弹性边界

不适用 / 改用（判断依据见 `../../reference/methods.md`）：
- **领域系统**（运营无关，如搜索引擎 / CMS）→ 对象建模 / 知识消化 / 四色
- 只需找领域事件 / 凭证链（不涉及合同履约）→ `bm-four-color`

## 前置：先判别领域系统 vs 业务系统

- 换**运营方式**（如加企业版）会变的功能 = 业务逻辑 → 用 8X Flow
- 换**内容 / 载体**（红薯→土豆、网文→网剧）仍成立的赚钱逻辑 = 业务逻辑
- 运营无关、描述某个问题域的 = 领域逻辑 → 用对象建模

不确定时默认：与盈利相关就用 8X Flow。

## 元模型（核心概念，详见 `../../reference/concepts.md`）

- **合同上下文**：最小业务上下文，只在两方之间（多方 = 多份双方合同），含**两个角色**
- **履约请求**：一个**时间段**，由权利方发起，要求义务方在规定时限内完成
- **履约确认**：一个**时间点**
- **凭证**：履约请求 = 发起依据，履约确认 = 履约证明；在权责范围内**按四色建模寻找**
- **标的物**：凭证涉及的对象（如商品），来自领域系统

Request-Confirmation 是**异步结构**——这是业务的真实状态，不是技术选择。履约失败触发新的履约项，而非自动修正。

## 五步建模工作流（增量推进，每步对应 report.html 章节）

启动 live server（见 `../../reference/live-session.md`）后，按以下五步逐章增量产出——每步问关键问题 → 更新 report.html → 浏览器自动刷新 → 用户确认 → 下一步：

```
8X Flow 进度（每步产出一章，增量追加进 report.html）：
- [ ] 1. 找合同上下文，明确参与方 → 合同上下文图
- [ ] 2. 找主要履约项，按四色找凭证 → 主要履约项表
- [ ] 3. 找违约情况，设新履约项，按四色找凭证 → 违约流转图
- [ ] 4. 重复步骤 3，直到不得不打官司为止 → 违约流转图（补全）
- [ ] 5. 把参与方与标的物划入领域边界 → 弹性边界 / 服务图
```

**步骤 1：找合同上下文，明确参与方**　合同只在两方之间；涉及多方时拆成多份双方合同。**平台 / 枢纽型业务**（外卖、电商、打车）判断枢纽方出现在哪些合同：识别每个盈利流的实际收款方 / 付款方——他们就是结算合同的对手方；只撮合不直接收付的是支撑方，不进核心合同（见 `../../reference/concepts.md` 的"枢纽参与方 / 平台型业务"）。

**步骤 2：找主要履约项，按四色找凭证**　先按"围绕该合同核心对价 / 收入流的履约项即主要履约项"筛选（见 `../../reference/concepts.md` 的"主要履约项 vs 违约履约项"）。对每项权责明确权利方 / 义务方；履约请求 = 时间段，履约确认 = 时间点。**找凭证的最小子步骤**（四色法核心）：① 围绕该履约项找现金 / KPI 凭证；② 列关键数据项（时间点 + 金额）；③ 追溯每个关键数据项的来源。若需完整凭证链，调用 `bm-four-color`。

**步骤 3：找违约情况，设新履约项**　对每个主要履约项寻找违约场景。判别处理方式（见 `../../reference/concepts.md` 的"独立履约项 vs 自动作废"）：违约需对方**主动作为**（退钱、赔付）→ 设独立履约项，可继续递归；违约只是**状态翻转**（如订单关闭）且无需对方确认 → 自动作废，到此终止。违约责任也需合同预先覆盖。按四色找凭证。

**步骤 4：重复步骤 3，直到不得不打官司**　违约的承担继续违约 → 再触发新履约项 → 最终诉诸法律（终极履约）即为**终止条件**。

**步骤 5：划领域边界**　把参与方与标的物划入领域边界（标的物常来自领域系统，通过凭证引用参与业务）。

> 示例（专栏订阅合同，参与方：读者 ↔ 极客时间）：主要履约项 = 支付订阅费（权利方极客时间）+ 访问付费内容（权利方读者）。违约：未限时支付 → 合同自动作废（无需独立权责项）；断更 → 退钱 + 重上架免费，若拒不退钱 → 只能打官司（终止）。

## 引入变化点（8X Flow 的核心价值）

跨合同上下文的凭证引用 = 上下文间的**依赖关系**。**核心合同依赖支撑合同**（如专栏订阅依赖各种支付合同）是坏味道——让核心逻辑依赖支撑逻辑。

解法：**反转依赖——让履约确认角色化**（能力供应商模式），让其他合同中的凭证来扮演这个履约确认，从而在核心合同中引入**变化点**。

> 示例：把"专栏付款确认"角色化，由移动支付合同 / 预充值合同等不同合同的凭证来扮演 → 一处变化点支持多种支付方式，核心合同不再逐个依赖新支付合同。

## 弹性边界落点

- 合同上下文 **≠** 弹性边界（同一合同的不同履约项弹性诉求不同，如访问内容比支付容量诉求大）
- **履约上下文 = 弹性边界**
- 领域逻辑边界也是弹性边界

依此切分领域与业务边界，即可应对云时代建模的四挑战（反映弹性边界、切分后维护一致性、异步解读业务、异步异常维护一致性）。**映射到服务的启发式**：按履约项的容量 / 频次聚类成服务（如高频大容量的"配送履约"独立，低频批处理的"结算履约"独立），而非按合同整体切。

## 补充：合同前与无合同场景（第 16 讲）

- **渠道上下文（合同前）**：邀请投标（RFP）→ 投标（Proposal）→ 签约 → 履约。渠道上下文是系统**第二大变化点**，弹性诉求与履约差距大，必须与履约上下文分离。
- **无对外合同的四种情况**：① 内部绩效系统（用绩效协议 + 进度检查建模，权责 = KPI/OKR）；② 纯领域系统（按领域建模）；③ 工具（胶水代码）；④ 你没意识到其中有合同（重新建模，把隐藏合同找出来）。

## 产出（HTML 报告为主，终端摘要）

按 `../../reference/output-format.md` + `../../reference/live-session.md`：**HTML 报告含完整方案**（图/表/判别，live server 实时预览），**终端每章只给一两句摘要**（不重复完整方案，省 token）。内容项（写在 HTML 里）都含：

1. **合同上下文图**：参与方 + 双方合同，**标出平台 / 枢纽方的对手方结算合同**（粗线）与仅撮合的合同区分。
2. **主要履约项表**：每项权利方 / 义务方 + 履约请求（时间段）/ 履约确认（时间点）+ 凭证。
3. **违约流转图**：每个违约 → 独立履约项 vs 自动作废的判别路径，终止于打官司。
4. **变化点清单**：角色化的履约确认、不同渠道上下文、不同标的物来源。
5. **弹性边界 / 服务图**：按履约上下文划分弹性边界 → 服务映射（区分业务系统 / 领域系统）。
6. **端到端走查**：正常流程 + 至少 1 个违约场景，验证模型自洽。

**章节结构是默认推荐、可按业务增删/重排**（标号 CSS counter 自动）；看完整效果用 live server 实时生成（见 `../../reference/live-session.md`），不必预读 examples。
