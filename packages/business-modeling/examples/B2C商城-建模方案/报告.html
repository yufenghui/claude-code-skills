场景假设

四色建模高度依赖业务模式假设。本方案采用 B2C 商城的主流形态。

维度假设对模型的影响
经营形态平台型（自营 + 第三方商家入驻），如天猫 / 京东 POP平台是枢纽方，存在抽佣结算
支付方式用户在线支付给平台（平台代收）引入担保交易、货款释放环节
物流配送商家发货，外包承运商配送（三通一达 / 顺丰）物流是独立合同段
逆向流程支持七天无理由退货电商特有逆向履约凭证
结算机制担保交易：确认收货后平台才放款给商家（扣佣）收货确认 = 货款释放触发点

端到端业务时序

凭证随业务推进依次产生；粗线 = 平台直接收付的结算。

入口识别

现金往来 → 收入流；无现金环节 → 成本结构（KPI）。

#入口类型性质
1用户支付订单货款（平台代收）现金收入流
2平台向商家结算货款（扣佣金 / 技术服务费）现金支出收入流
3平台 / 商家向承运商支付物流费现金支出收入流
4发货时效（商家承诺 N 小时内发货）无直接现金KPI（成本结构）
5配送时效（承诺 N 天送达）无直接现金KPI（成本结构）

业务脊梁：凭证链

实线 = 数值 / 时标来源；虚线 = 规则来源。

凭证表

异常 / 逆向凭证

逆向流程也是凭证，是权责追溯的关键考验。

逆向场景凭证判定
超时未支付订单关闭自动作废（状态翻转，无需对方确认）→ 终止
已付款未发货，用户取消Refund独立履约项（退钱），refund_at > paid_at
已收货，七天无理由Return + Refund受 ReturnPolicy.return_window 时标约束
商家发货超时TimeoutPenaltyKPI 偏差触发（扣保证金 / 降权 / 赔付）

KPI 凭证链

电商有双段物流时效（外卖只有一段）：发货时效约束商家，配送时效约束承运商。

偏差追溯：实际时点与目标 deadline 对比，超出触发 TimeoutPenalty（扣商家保证金 / 降权 / 扣承运商运费），进入结算合同的违约处理。

四原型分布

原型颜色本场景元素
时标型（Moment-Interval）红/粉Order、Payment、ShipmentConfirm、DeliveryConfirm、MerchantSettlement、LogisticsFee、Refund、Return、ShipPromise、DeliveryPromise、TimeoutPenalty
描述型（Description）蓝SKU 商品目录、PromotionRule、Coupon、FreightRule、CommissionRule、ReturnPolicy
角色型（Role）黄买方、卖方、付款方、收款方、发货方、承运方
参与者型（Party-Place-Thing）绿用户、商家、平台、承运商、商品 (SKU)（标的物）

关键判别应用说明

  1. 担保交易时标（电商独有）：MerchantSettlement.settled_at > DeliveryConfirm.received_at。平台代收货款，确认收货触发放款——区别于外卖"送达即结算"，是 B2C 凭证链的核心差异点。
  2. 价格快照锁定：Order 在 placed_at 锁定 SKU 价格快照与库存，商品后续变价不溯及既有订单。SKU 目录归描述型，其快照内嵌进 Order。
  3. 算法生成金额（关键数据项第 ③ 类来源）：物流费 fee = f(重量, 地区, 满减) 追溯到描述型 FreightRule；促销折扣追溯 PromotionRule。
  4. 描述型 vs 时标型：促销规则、优惠券虽影响金额，但仅被订单引用、承载规则 → 描述型。
  5. 标的物归类：商品 (SKU) = 标的物 → 参与者型（Party-Place-Thing，含"物"），非第五类；业务上下文仅以 ID / 快照引用，不拥有其生命周期。
  6. 平台枢纽方：平台在"货款结算合同""物流费结算合同"中作为对手方直接收付（盈利来源 = 抽佣），在"订单交易"中仅撮合。
  7. 双段物流 KPI：发货时效（约束商家）+ 配送时效（约束承运商）两条独立 KPI 链，责任分摊清晰。

端到端走查

✅ 正常：浏览 → 下单 → 付款 → 发货 → 收货 → 结算

  1. 用户浏览 SKU 目录（描述型）→ 下单 Order（placed_at，锁定价格快照与库存，total_amount 由 SKU 快照价 − 优惠计算）。
  2. 用户在线支付 → Payment（paid_at > placed_at，amount = Order.total_amount），平台代收。
  3. 商家发货 → ShipmentConfirm（shipped_at > paid_at，受 ShipPromise.deadline 约束）；承运商配送 → 用户收货 → DeliveryConfirm（received_at，受 DeliveryPromise.deadline 约束）。
  4. 担保放款：received_at 后 → MerchantSettlement（settled_at > received_at，amount = Σ(Order)×(1−rate) − Refund）；同时 LogisticsFee 结算承运商。

⚠ 异常 A：下单后变价（验证价格快照）

商品 14:00 从 199 元调为 219 元；用户 13:59 下单 → 订单按旧价 199（快照锁定），Order 不受 SKU 后续变更影响。✅ 脊梁可证明。

⚠ 异常 B：收货后七天无理由退货（验证逆向凭证）

received_at 为 D 日，用户 D+5 日申请退货（≤ received_at + return_window=7）→ Return → Refund（refund_at > paid_at，引用 Payment）→ 从下次 MerchantSettlement 扣减。✅ 权责可追溯。

⚠ 异常 C：商家发货超时（验证 KPI 偏差）

shipped_at > ShipPromise.deadline → 触发 TimeoutPenalty（扣保证金 / 降权）；若用户取消 → Refund 仅退款。✅ 脊梁经受追溯。

自洽性结论

✅ 两条硬约束满足
所有凭证围绕现金往来（货款 / 结算 / 物流费）或 KPI（发货 / 配送时效）；凭证间通过关键数据项（时点 + 金额）的计算逻辑与时标顺序逻辑明确关联。
✅ 脊梁完整且经异常验证
业务脊梁 Order → Payment → ShipmentConfirm → DeliveryConfirm → MerchantSettlement 完整捕捉 B2C 收入流，并经变价 / 退货 / 超时三类异常追溯验证。
