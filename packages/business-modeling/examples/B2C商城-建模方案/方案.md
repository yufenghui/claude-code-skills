## 0. 场景假设（需确认）

四色建模高度依赖业务模式假设。本方案采用 B2C 商城的主流形态：

| 维度 | 假设 | 对模型的影响 |
| --- | --- | --- |
| 经营形态 | **平台型**（自营 + 第三方商家入驻），如天猫 / 京东 POP | 平台是枢纽方，存在抽佣结算 |
| 支付方式 | 用户**在线支付**给平台（平台代收） | 引入担保交易、货款释放环节 |
| 物流配送 | 商家发货，**外包承运商**配送（三通一达 / 顺丰） | 物流是独立合同段 |
| 逆向流程 | 支持**七天无理由退货** | 电商特有逆向履约凭证 |
| 结算机制 | **担保交易**：确认收货后平台才放款给商家（扣佣） | 收货确认 = 货款释放触发点 |

> 与外卖的关键差异：外卖是"即时送达即结算"，B2C 是"**担保交易 + 账期 + 确认收货触发结算**"。

## 1. 端到端业务时序

凭证随业务推进依次产生：

```
用户            平台             商家            承运商
 |               |                |                |
 |--下单-------->|                |                |
 |--发起支付---->|                |                |
 |               |--支付确认----->| (自身)         |
 |               |--派单--------->|                |
 |               |                |--备货--------->|
 |               |                |--发货--------->|
 |               |                |       ...送达/收货确认
 |<==============货款结算(扣佣)===================|
 |               |===========物流费结算==========>|

Note: paid_at > placed_at；settled_at > received_at（担保放款）
```

## 2. 入口识别

| # | 入口 | 类型 | 性质 |
| --- | --- | --- | --- |
| 1 | 用户支付订单货款（平台代收） | 现金 | 收入流 |

| 5 | 配送时效（承诺 N 天送达） | 无直接现金 | KPI（成本结构） |

## 3. 业务脊梁：凭证链

```
凭证链（--> 数值/时标来源）

  [Order 订单] --> [Payment 支付] --> [Shipment 发货] --> [Delivery 收货] --> [Settlement 结算]
   placed_at       paid_at           shipped_at          received_at        扣佣·担保放款
   total_amount    amount
       |
       +--> [LogisticsFee 物流费]   fee = f(重量, 地区)

规则来源（..> 引用）

  [SKU 商品目录]   ..> [Order]
  [优惠券 Coupon]  ..> [Order]
  [Promotion 促销] ..> [Order]
  [FreightRule]    ..> [LogisticsFee]
  [CommissionRule] ..> [Settlement]
  [ReturnPolicy]   ..> [Delivery]
```

时标顺序逻辑（关键约束）：

```
  paid_at     > placed_at        先下单后支付
  shipped_at  > paid_at          先付后发（COD 是变化点）
  received_at > shipped_at
  settled_at  > received_at      担保交易：收货确认触发放款
```

### 凭证表

| 凭证 | 原型 | 关键数据项 | 来源 / 计算逻辑 | 时标顺序逻辑 |
| --- | --- | --- | --- | --- |
| Order 订单 | 时标型 | `placed_at`, `total_amount` | `total_amount = Σ(SKU快照价 × qty) − Coupon − Promotion` | `placed_at` 锁定价格快照，变价不溯及既有订单 |
| Payment 支付 | 时标型 | `paid_at`, `amount` | `amount = Order.total_amount` | `paid_at > placed_at` |
| ShipmentConfirm 发货 | 时标型 | `shipped_at` | KPI"实际发货"凭证 | `shipped_at > paid_at` |
| DeliveryConfirm 收货 | 时标型 | `received_at` | 用户确认 / 超时自动确认 | `received_at > shipped_at` |
| MerchantSettlement 商家结算 | 时标型 | `settled_at`, `amount` | `amount = Σ(Order.total_amount) × (1 − commission_rate) − Σ(Refund)` | **`settled_at > received_at`**（担保放款） |
| LogisticsFee 物流费 | 时标型 | `paid_at`, `fee` | `fee = f(重量, 地区, 满减规则)`（算法计算，追溯 FreightRule） | 账期内 |
| Refund 退款（逆向） | 时标型 | `refund_at`, `amount` | `amount = Payment.amount` | `refund_at > received_at`；引用 Payment |
| Return 退货（逆向） | 时标型 | `returned_at` | 退货物流确认 | `returned_at ≤ received_at + return_window` |
| SKU 商品目录 | 描述型 | `price`, `spec` | 规则 / 目录模板，被订单引用快照 | — |
| PromotionRule / Coupon | 描述型 | 算法 / `face_value` | 规则模板 | — |
| FreightRule / CommissionRule / ReturnPolicy | 描述型 | 算法 / `rate` / `return_window` | 规则模板 | — |
| **商品 (SKU)** / 用户 / 商家 / 平台 / 承运商 | **参与者型** | — | **标的物 = 商品 (SKU)**，业务上下文不拥有其生命周期 | — |
| 买方 / 卖方 / 付款方 / 收款方 / 发货方 / 承运方 | 角色型 | — | 参与方在凭证中的角色 | — |

## 4. 异常 / 逆向凭证

```
[Order 已下单]
   |
   |-- 超时未支付 ----------> (自动作废) 订单关闭 --> [终止]
   |-- 已付款未发货取消 ----> [Refund 仅退款]        (独立履约项)
   |-- 已收货·七天无理由 --> [Return + Refund]      (受 return_window 约束)
   |-- 发货超时(KPI偏差) --> [TimeoutPenalty 罚扣]  (扣保证金 / 降权)
   |                              |
   |                              +-- 拒不赔付 --> [打官司 --> 终止]
```

| 逆向场景 | 凭证 | 判定 |
| --- | --- | --- |
| 超时未支付 | 订单关闭 | **自动作废**（状态翻转，无需对方确认）→ 终止 |
| 已付款未发货，用户取消 | `Refund` | 独立履约项（退钱），`refund_at > paid_at` |
| 已收货，七天无理由 | `Return` + `Refund` | 受 `ReturnPolicy.return_window` 时标约束 |
| 商家发货超时 | `TimeoutPenalty` | KPI 偏差触发（扣保证金 / 降权 / 赔付） |

## 5. KPI 凭证链

电商有**双段物流时效**（外卖只有一段配送）：发货时效约束商家，配送时效约束承运商。

```
[ShipPromise 目标 deadline]  ..对比..  [ShipmentConfirm 实际 shipped_at]
                                                |
                                      shipped_at > deadline ?
                                          |-- 是 --> [TimeoutPenalty 罚扣]

[DeliveryPromise 目标 deadline] ..对比.. [DeliveryConfirm 实际 received_at]
                                                |
                                      received_at > deadline ?
                                          |-- 是 --> [TimeoutPenalty 罚扣]
```

偏差追溯：实际时点与目标 `deadline` 对比，超出触发 `TimeoutPenalty`（扣商家保证金 / 降权 / 扣承运商运费），进入结算合同的违约处理。

## 6. 四原型分布

| 原型 | 颜色 | 本场景元素 |
| --- | --- | --- |
| **时标型**（Moment-Interval） | 红/粉 | Order、Payment、ShipmentConfirm、DeliveryConfirm、MerchantSettlement、LogisticsFee、Refund、Return、ShipPromise、DeliveryPromise、TimeoutPenalty |
| **描述型**（Description） | 蓝 | SKU 商品目录、PromotionRule、Coupon、FreightRule、CommissionRule、ReturnPolicy |
| **角色型**（Role） | 黄 | 买方、卖方、付款方、收款方、发货方、承运方 |
| **参与者型**（Party-Place-Thing） | 绿 | 用户、商家、平台、承运商、**商品 (SKU)（标的物）** |

---

## 7. 关键判别应用说明

1. **担保交易时标（电商独有）**：`MerchantSettlement.settled_at > DeliveryConfirm.received_at`。平台代收货款，**确认收货触发放款**——区别于外卖"送达即结算"，是 B2C 凭证链的核心差异点。
2. **价格快照锁定**：`Order` 在 `placed_at` 锁定 SKU 价格快照与库存，商品后续变价不溯及既有订单。SKU 目录归**描述型**，其快照内嵌进 Order。
3. **算法生成金额（关键数据项第 ③ 类来源）**：物流费 `fee = f(重量, 地区, 满减)` 追溯到**描述型** `FreightRule`；促销折扣追溯 `PromotionRule`。
4. **描述型 vs 时标型**：促销规则、优惠券虽影响金额，但仅被订单引用、承载规则 → **描述型**。
5. **标的物归类**：**商品 (SKU)** = 标的物 → **参与者型（Party-Place-Thing，含"物"）**，非第五类；业务上下文仅以 ID / 快照引用，不拥有其生命周期。
6. **平台枢纽方**：平台在"货款结算合同""物流费结算合同"中作为**对手方**直接收付（盈利来源 = 抽佣），在"订单交易"中仅撮合。

## 8. 端到端走查

### 8.1 正常：浏览 → 下单 → 付款 → 发货 → 收货 → 结算

1. 用户浏览 SKU 目录（描述型）→ 下单 `Order`（`placed_at`，锁定价格快照与库存，`total_amount` 由 SKU 快照价 − 优惠计算）。
2. 用户在线支付 → `Payment`（`paid_at > placed_at`，`amount = Order.total_amount`），平台代收。
3. 商家发货 → `ShipmentConfirm`（`shipped_at > paid_at`，受 `ShipPromise.deadline` 约束）；承运商配送 → 用户收货 → `DeliveryConfirm`（`received_at`，受 `DeliveryPromise.deadline` 约束）。
4. 担保放款：`received_at` 后 → `MerchantSettlement`（`settled_at > received_at`，`amount = Σ(Order)×(1−rate) − Refund`）；同时 `LogisticsFee` 结算承运商。

### 8.2 异常 A：下单后变价（验证价格快照）

商品在 14:00 从 199 元调为 219 元；用户 13:59 下单 → 订单按**旧价 199**（快照锁定），`Order` 不受 SKU 后续变更影响。**✅ 脊梁可证明。**

### 8.3 异常 B：收货后七天无理由退货（验证逆向凭证）

`received_at` 为 D 日，用户 D+5 日申请退货（≤ `received_at + return_window=7`）→ `Return`（退货物流）→ `Refund`（`refund_at > paid_at`，`amount = Payment.amount`，引用 Payment）→ 该 Refund 从商家下次 `MerchantSettlement` 中扣减。**✅ 权责可追溯。**

### 8.4 异常 C：商家发货超时（验证 KPI 偏差）

`shipped_at > ShipPromise.deadline` → 触发 `TimeoutPenalty`（扣商家保证金 / 降权）；若用户选择取消 → `Refund` 仅退款。**✅ 脊梁经受追溯。**

---

## 9. 自洽性结论

- ✅ **两条硬约束满足**：所有凭证围绕现金往来（货款 / 结算 / 物流费）或 KPI（发货 / 配送时效）；凭证间通过关键数据项（时点 + 金额）的**计算逻辑**与**时标顺序逻辑**明确关联。
- ✅ 业务脊梁 `Order → Payment → ShipmentConfirm → DeliveryConfirm → MerchantSettlement` 完整捕捉 B2C 收入流，并经变价 / 退货 / 超时三类异常追溯验证。

## 10. 待确认的业务假设

1. **是否担保交易？** 若是京东自营（平台 = 卖方，无账期结算），则 `MerchantSettlement` 与抽佣环节消失。
2. **是否货到付款（COD）？** 若支持，`shipped_at > paid_at` 时标失效，支付时序需独立变化点。
3. **物流是否自营？** 若自营（如京东物流），承运商并入平台，`LogisticsFee` 结算对象变化。
4. **是否含预售 / 众筹？** 会引入"定金 + 尾款"双支付凭证链。
