## 一、总纲：业务建模与 DDD

### 业务建模（Business Modeling）
- **是什么**：一套**定义问题**的方法（其次才是解决问题）。准确界定业务问题，往往能把解决方案复杂度降几个数量级（例：PDF 渲染集群 → 一条 if 语句）。
- **两个难点**：① 定义问题并让所有干系人接受；② 在特定架构约束下把模型实现出来。
- **出处**：开篇词。

### 知识消化 / 两关联一循环（Knowledge Crunching）
- **是什么**：Eric Evans 在 DDD 中提倡的提炼领域模型的方法，是 DDD 的核心过程。
- **两关联**：模型 ↔ 软件实现关联；统一语言 ↔ 模型关联。
- **一循环**：提炼知识的循环（用统一语言讨论需求 → 发现模型缺失/不当 → 精炼模型 → 改变统一语言 → 试验验证 → 循环）。
- **判断"是否在做 DDD"**：核心是**模型与软件实现是否关联**（修改模型=修改代码）。不配合统一语言与提炼循环，就只是一种编码风格。
- **出处**：01、02、03。

### 统一语言（Ubiquitous Language）
- **是什么**：从领域模型提取的、业务与技术共用的共同语言，是知识消化的协同枢纽。
- **关键区分**："统一语言提案" ≠ "统一语言"——只有多方认可、集体所有后才真正成立。
- **本质**：统一语言是**共识的载体**；没有共识，词汇毫无意义（有共识则用什么词都行，当然要与模型关联）。出处：结束语。
- **出处**：02、07、08、结束语。

### 分析模型 vs 设计模型
- **是什么**：当模型与实现割裂，模型会分裂成更接近业务的分析模型（沟通需求用）与更接近实现的设计模型。
- **判断**：一旦出现这种分裂，统一语言与提炼知识循环就不再发生——这是知识消化失败的前兆。
- **出处**：01。

## 二、DDD 的实现模式（"跨越现实的障碍"）

### 贫血模型 vs 充血模型（富含知识的模型，Knowledge Rich Model）
- **贫血**：对象只封装简单数据，关联关系与业务计算散落在对象之外（DAO/存储过程），沿用过程式风格。
- **充血/富含知识**：与某概念相关的主要行为与逻辑，封装进对应领域对象。
- **判断**：业务逻辑富集于**聚合根**，而非散落于 DAO。
- **出处**：01、05。

### 聚合 / 聚合根（Aggregation / Aggregation Root）
- **是什么**：通过聚合关系连接的对象，从概念上是一个整体；聚合根是必不可少的对象，管理其他对象的生命周期。
- **判断**：若把对象拆开就无法表达业务概念，则它们构成聚合。如"用户订阅的专栏"= User（聚合根）+ Subscription（不可脱离 User 独立存在）。
- **出处**：01。

### 关联对象（Association Object）
- **是什么**：把对象间的关联关系显式建模为一个接口对象（如 `MySubscriptions extends Iterable`），承载分页/计算/排序等集体逻辑，隔离数据库/网络开销。
- **问题信号**：聚合里直接用 `List`/集合 → 陷入"全量加载 vs N+1 vs 逻辑泄露"三难。
- **出处**：04。

### 角色对象（Role Object）/ 上下文对象（Context Object）
- **是什么**：解决"上下文过载"（一个实体在多个上下文当聚合根 → 过大类坏味道）。角色对象把 Reader/Buyer 等做成装饰器，经 Repository 的 `asXxx(user)` 显式切换上下文；上下文对象把界限上下文本身建模成接口。
- **判断信号**：一个 User 在多处当聚合根、改不动 = 模型僵硬。
- **出处**：05。

### 能力供应商（Capability Provider）
- **是什么**："基础设施不是层"。把技术组件**拟人化**为有业务含义的能力接口（如 `BankPaymentClient → Cashier`、`EmailClient → CustomerService`）放进领域层，基础设施实现它作为"供应商"。这是关联/角色/上下文对象的元模式，也是 8X Flow 引入变化点的手法（反转依赖）。
- **判断**：任何从技术组件提取的词，只要有业务含义且业务方认可，就是领域概念。
- **出处**：06、09（8X Flow 变化点复用它）。

## 三、四色建模法（4 Color Modeling）

### 强分析法 vs 弱分析法
- **弱分析（头脑风暴/事件风暴）**：发散-收敛获取事件流，收敛逻辑语焉不详，极度依赖主持人经验，"一学就会一用就废"。
- **强分析（四色法）**：从明确的收敛逻辑（业务底层逻辑）出发，直接引导-分析获取事件流，可经逻辑推演、不依赖灵感。
- **出处**：08、09。

### 现金日记账 / 总账 / 试算平衡表（财务视角）
- **日记账（Journal）**：出纳记录现金收付，记**关键事件**（现金结果）。
- **总账（General Ledger）**：按科目汇总，记事件的**缘起与后续**（过程单据）。
- **试算平衡表**：总账汇总而成。
- **洞察**：抓住日记账上的事件，就能围绕它建立前后关联的事件流。
- **出处**：09。

### 三条核心逻辑（四色法发现事件的依据）
1. **现金收入 → 承担义务**：拿钱办事，需收集证据证明义务履约成功。
2. **现金支出 → 拥有权利**：花钱消灾，需检查对方是否按时履约。
3. **无现金往来 → 目标-实际对比（KPI）**：设目标/计划，追踪实际，产生类似履约的约束。
- 前两条源自**收入流**（Revenue Stream），第三条源自**成本结构**（Cost Structure）。
- **出处**：09。

### 凭证（Evidence）/ 关键数据项 / 凭证链
- **凭证**：围绕现金往来或 KPI 的记录，关键数据项通常是**时间点 + 金额**。
- **凭证链**：凭证间通过关键数据项的**来源**明确关联（计算逻辑与时标顺序逻辑），串联成领域事件流。
- **两条硬约束**：① 凭证必须围绕现金往来或 KPI；② 凭证间必须通过关键数据项明确关联。
- **出处**：09。

### 业务脊梁（Backbone of the business）
- **是什么**：由关键数据项彼此关联的凭证链，捕捉一项业务的收入流全过程。
- **稳定性依据**：业务模式（Business Model）在企业生命周期内相当稳定，而业务模式主要通过收入流和成本结构体现——故从源头建模稳定部分。
- **出处**：09。

### 四原型（四色）
- **时标型（Moment-Interval，红/粉）**：凭证，带时间戳，是建模的锚点。
- **描述型（Description，蓝）**：凭证的类型/目录，承载共享属性（如商品目录、抽佣规则模板）。
- **角色型（Role，黄）**：参与方在某凭证中扮演的角色（如付款方、收款方）。
- **参与者型（Party-Place-Thing，绿）**：人 / 组织 / 地点 / **物**。**标的物（如商品、餐品）归此类**——它是被凭证引用的独立实体，不是独立的第五类。
- **描述型 vs 时标型判别**：若某对象**独立参与现金 / KPI 权责追溯**（自身是凭证链一环）→ 时标型；若**仅被引用**、承载共享属性（目录 / 规则模板）→ 描述型。报价单、优惠券虽带时间戳，但通常是被订单引用的规则，归描述型。
- **出处**：09。

### 角色扮演游戏验证
- **是什么**：把凭证转成真实单据，让业务方模拟经营（含异常追溯），验证业务脊梁能否证明权责。
- **出处**：09。

### 异常 / 逆向凭证（退款、取消等）
- **是什么**：收入流里的逆向流程（退款、取消、赔付）也是凭证，且是权责追溯最关键的考验场景——不能只建正向流程。
- **退款范例**：`Refund` 凭证，关键数据项 `refund_at`（时间点）+ `amount`；计算 `amount = Payment.amount`；时标 `refund_at > Payment.paid_at`；常引用原支付凭证作为退款依据。
- **出处**：09（异常追溯是"角色扮演验证"的重点）。

### KPI 凭证链结构（成本结构入口）
- **是什么**：无直接现金往来的环节，用"目标-实际对比"产生类似履约的约束。一条 KPI 链由三部分组成。
- **结构**：① **目标凭证**（带 deadline / 目标值，如配送时效承诺）；② **实际履约凭证**（如送达确认 `arrived_at`）；③ **偏差追溯**（实际 vs 目标，超出则触发扣款 / 差评等违约凭证）。
- **范例（配送时效）**：`DeliveryKPI`(deadline) → `DeliveryConfirm`(arrived_at) → 若 `arrived_at > deadline` 触发超时罚扣凭证。
- **出处**：09（原文 KPI 建模留作思考题，此处补全结构）。

### 履约凭证的发现规则
- **是什么**：对每条现金权责，问"付了钱得到了什么权利 / 收了钱承担了什么义务"，权利或义务对应一张履约凭证。原文只给了"订阅"一个孤例，此处补双侧规则。
- **现金支出侧**（花钱 → 拥有权利）：如平台付钱给骑手 → 拥有"获得配送服务"的权利 → 由送达凭证证明对方履约。
- **现金收入侧**（收钱 → 承担义务）：如读者付费 → 平台承担"提供阅读权"的义务 → 订阅凭证（`started_at`），约束其晚于支付完成。
- **出处**：09。

### 关键数据项的三类来源
- 追溯"关键数据项从哪来"时，来源有三类：① **前序凭证的数值**（如支付金额 ← 订单金额）；② **人工输入**（如编辑输入报价）；③ **由前序凭证中的规则 / 算法计算得到**（如骑手配送费 = f(距离, 时段, 天气)，追溯到 `RiderAgreement` 的计费规则）。
- **出处**：09。

## 四、云时代的架构约束

### 弹性边界 / 弹性优先原则
- **是什么**：云时代引入的新约束——弹性是最需优先考虑的因素。弹性边界是第一划分依据。
- **弹性优先判断**：把某业务上下文放入独立弹性边界，**能否利用云的弹性更易控制成本**。能→拆；不能→不拆。
- **出处**：12。

### 弹性依赖 vs 弹性耦合
- **弹性依赖（云友好）**：依赖对方的**吞吐量**。
- **弹性耦合（云不擅长）**：依赖对方的**响应时间**（上游扩容后下游必须同步等待）。
- **改造手法**：同步调用 → 异步（引入请求缓冲/队列），把响应时间依赖换算成吞吐量。
- **出处**：13。

### 领域逻辑 vs 业务逻辑
- **领域逻辑**：运营无关、领域特定，复杂度来自问题本身（如推荐引擎、CMS、搜索引擎）。
- **业务逻辑**：运营特定、领域中立，复杂度来自盈利流程（如定价、分成、广告）。
- **判别测试**：换运营方式（如加企业版）会变的功能=业务逻辑；换内容/载体（红薯→土豆、网文→网剧）仍成立的赚钱逻辑=业务逻辑。
- **出处**：14。

### 业务运营模型（IBM 四种）
- 统一模式（单实例）/ 复制模式（多租户 SaaS）/ 协作模式（共享 API）/ 差异化模式（开放生态）。云把最低成本点从"统一"推到"复制"。
- **出处**：14。

## 五、8X Flow（合同履约法）

### 合同（Contract）/ 权责履约（Fulfillment）
- **合同 = 最小业务上下文**：只在两方之间（多方=多份双方合同），故每个合同上下文有**两个角色**。
- **权责履约 = 最小业务交互**。
- **出处**：15。

### 履约请求 / 履约确认（Request-Confirmation）
- **履约请求**：表示一个**时间段**，由权利方发起，要求义务方在规定时限内完成；关联凭证作为发起依据。
- **履约确认**：一个**时间点**，关联凭证作为履约证明。
- **Request-Confirmation 是异步结构**：未确认前履约处于未知状态——这是业务的真实状态，不是技术选择。
- **出处**：15。

### 凭证 / 标的物
- 凭证在权责履约范围内**按四色建模寻找**；履约请求/确认本身也是凭证，可被引用。
- **标的物**：凭证中涉及的对象（如商品），可能来自领域系统，通过凭证引用参与业务逻辑。
- **出处**：15。

### 违约与终止条件
- 违约触发**新的履约项**（在合同上下文中维持一致性）；对违约的承担继续违约 → 再触发 → 最终**打官司**（终极履约）。
- **建模终止条件**：重复找违约履约项，"直到不得不打官司为止"。
- **出处**：15。

### 变化点 / 履约确认角色化
- **问题**：核心合同依赖支撑合同（如专栏订阅依赖各种支付合同）是坏味道——核心逻辑依赖支撑逻辑。
- **解法**：反转依赖——**让履约确认角色化**（能力供应商模式），让其他合同中的凭证来扮演它 → 在核心合同中引入**变化点**。
- **出处**：15。

### 弹性边界（8X Flow 视角）
- 合同上下文 ≠ 弹性边界（不同履约项弹性诉求不同）；**履约上下文才是弹性边界**；领域逻辑边界也是弹性边界。
- **出处**：15。

### 渠道上下文（合同前）
- 合同签订之前：邀请投标（RFP）→ 投标（Proposal）→ 签约 → 履约。合同前的"渠道上下文"是系统第二大变化点，弹性诉求与履约差距大，必须分离。
- **出处**：16。

### 主要履约项 vs 违约履约项
- **主要履约项**：围绕该合同**核心对价 / 核心收入流**的履约项（如订单合同的"支付餐费""提供餐品"）。
- **违约履约项**：主要履约项失败触发的补偿 / 惩罚履约项（如退款、赔付）。
- **出处**：15。

### 独立履约项 vs 自动作废（违约处理判别）
- **独立履约项**：违约后果需要违约方**主动作为**（退钱、赔付、补做）→ 设独立履约项，可继续违约递归。
- **自动作废**：违约后果只是**状态翻转**（如订单关闭）且无需对方确认 → 无独立权责项，到此终止。
- **出处**：15。

### 标的物的引用形态
- 凭证以**标的物 ID / 快照**引用领域对象（如订单引用商品 ID），业务（合同）上下文**不拥有**标的物的生命周期——标的物归领域系统。
- **出处**：15。

### 枢纽参与方 / 平台型业务
- **是什么**：多方业务中常有贯穿所有合同的枢纽方（如外卖的"平台"、打车的"平台"、电商的" marketplace"）。
- **判断它出现在哪些合同**：识别每个盈利流的**实际收款方 / 付款方**——他们就是结算合同的对手方；**只撮合、不直接收付**的是支撑方，不进核心合同（但可作为被引用的领域 / 支撑系统）。
- **平台型示例（外卖）**：用户↔商家（订单合同）、用户↔支付方（支付协议）、商家↔平台（抽佣结算）、骑手↔平台（配送费结算）；平台在两份结算合同中作为对手方出现（盈利来源），在订单合同中仅撮合。
- **出处**：14、15（原文为非典型平台，此处补平台型规则）。

## 六、中台 / 微服务 / SaaS（供后续 skill 复用）

### 中台（Middle Platform）/ 宏流程（Macro-Process）
- **中台是什么**：支撑多个**小而灵活的前台团队**的软件平台。关键判别：中台服务的"前台"不是软件，而是**具有自主性的团队**（如按城市、按品类划分的独立运营团队）——这是理解中台与普通平台差异的入口。
- **前台团队的诉求**：高效重用（支撑业务快速扩张，如进新城、上新品类）+ 高度自主自治（适应各自业务环境）。中台 = 同时满足这两者的软件系统。
- **效能 vs 自主的平衡**：中台是软件平台，需平衡"效能下限"（**平台能力**，越丰富则应用开发成本越低）与"创新上限"（**应用自主度**，越高则创新空间越大）。中台把平衡点放在"特定场景中的业务模式"。
- **中台 vs SaaS**：① SaaS 是前端到后台全部系统，中台只有后端部分（**BaaS，后端即服务**），不新增前端渠道 / 入口；② 中台是"**产生 SaaS 的 SaaS**"，不直接服务租户，而是产出针对某业态的后台 SaaS。
- **宏流程 = 特定场景中的业务模式**：把多个类似业务模式的前台抽象后，提炼出的共有底层"核心业务模式"。业务上 = 宏观流程（不指某个前台的具体流程）；技术上 = 业务流程的**模板**，靠**配置点 + 扩展点**实例化成具体流程（实例化即 SaaS 化）。例：交易中台承载的交易业务模型、出行中台承载的"撮合 + 履约"业务模型。
- **建模宏流程（关键）**：从**已存在的 8X Flow 业务模型**中**提取**，而非直接建模——直接建模极易过度抽象、丧失对业务决策的指导意义。提取方法是 8X Flow 的延伸，步骤见 `bm-8x-flow` skill「中台化：宏流程提取」延伸章节；提要见 `methods.md` §7。
- **硬约束**：提取宏流程时，只可对**履约确认项**角色化引入变化点，**不可对履约请求项**角色化（破坏权责语义）。
- **出处**：17（中台定义、效能 / 自主、宏流程）、18（从合同上下文提取宏流程、角色化约束）。
- **真微服务 vs 伪微服务**：真微服务按业务能力划分、以产品研发、逻辑集中编排简单；伪微服务四反模式：分布式服务/微工作组/傻服务/分布式单体。（19）
- **选项（Offering）= 能力 + 服务质量（SLA/NFR）**：SaaS 化要围绕"选项"而非"服务"。（20）
- **魔球建模法**：把客户分三类设计选项——高价值（重响应，独立组网）/ 现金奶牛（覆盖大多数，降成本）/ 尷尬（价值低成本降不下，尽量转化掉）。（20）
