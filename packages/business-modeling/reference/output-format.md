# 建模产出规范

所有业务建模 skill 的产出都应遵循本规范，保证产出**可视化、可核查、结构一致**。"业务建模至少要有一张能看的图"——纯表格或纯文字不算合格交付。

## 视角纪律：业务为主角，方法为隐形工具（重要）

报告的主角永远是**被建模的业务**，不是建模方法本身。方法是隐形工具——像摄影师的相机，不该出现在照片里。产出遵循三条禁令：

1. **不介绍 / 夸赞方法**：报告正文不出现对方法本身的美化或导论（如"本方法借鉴职业经理人视角""强分析法优于事件风暴"）。方法背景只属于 skill 内部，不进交付物。
2. **不写 skill 评测视角**：报告是交付业务方的**业务模型**，不是 skill 的自测报告——禁出现"核查 / 卡点 / 重测 / 验证理论 / 排除水分 / 交叉验证"等 QA 词汇。
3. **不用业务佐证理论**：用方法**解析**业务、把结论落在业务上（"该业务的结算对手方是平台，因为平台直接收付抽佣"），而不是拿业务当素材去证明方法巧妙（"这个例子验证了变化点的威力"）。

**判别准则的表达**：写成**业务结论的依据**，不是**理论的注脚**。
- ✅ 对："餐品归参与者型，因为它是被凭证引用的标的物（Party-Place-Thing 含'物'）。"
- ❌ 错："此处应用了四色法参与者型判别准则。"

## 穷尽纪律：MECE（重要）

业务建模的产出必须**相互独立、完全穷尽**（MECE）。用户未声明忽略的维度，不得以"一概而过"或"挑几个示意"搪塞——每个维度都要逐项过、穷尽到判据满足。

5 个维度的穷尽判据（8X Flow 视角；判据定义详见 `concepts.md` 对应条目）：

| 维度 | 穷尽判据 |
| --- | --- |
| **上下文穷尽** | 对照每个盈利流的实际收付方找全合同；跑「无合同四场景」前置判别，确认无隐藏合同遗漏 |
| **履约项穷尽** | 每个上下文的主要履约项覆盖核心对价**两侧**：收入流侧（收钱承担的义务）+ 成本结构 / KPI 侧（花钱拥有的权利） |
| **违约穷尽** | 对**每个**主要履约项逐项找违约（非挑几个）；逐项标注「独立履约项 vs 自动作废」，递归到打官司终止 |
| **变化点穷尽** | 覆盖**两个来源**：① 角色化的履约确认；② 渠道上下文 |
| **边界穷尽** | 弹性边界覆盖：履约上下文 + 领域上下文 + 渠道上下文（合同上下文为服务边界，非弹性边界） |

> 与「视角纪律」一样，穷尽判据要写成**业务结论的依据**（"该上下文的结算对手方是平台，因平台直接收付抽佣"），不是理论的注脚。四色法视角同理：穷尽所有现金 / KPI 入口、穷尽异常 / 逆向追溯。

## 双轨产出（重要）

业务建模的产出分**两轨**，缺一不可：

| 轨 | 载体 | 图表形式 | 用途 |
| --- | --- | --- | --- |
| **终端轨** | Markdown（直接在对话/终端渲染） | **纯 ASCII 图**（`+ - | > v`，不用 Mermaid） | 快速对齐、任何终端/IDE 可读、可粘进 issue/wiki |
| **报告轨** | HTML 文件（交付业务方） | **手写 SVG**（PPT 级、离线矢量） | 正式交付、统一语言提案、可打印 |

> **为什么不用 Mermaid**：终端/对话里 Mermaid 代码块不会渲染成图，等于死代码。终端轨用纯 ASCII 当场可读；需要精美图时出 HTML 报告用手写 SVG。两轨的图**语义一致**（同样的节点、线型、时标约束），只是渲染介质不同。

## 必含内容（两轨都要有）

1. **结构化表格**：凭证 / 合同 / 履约等关键字段用表格列清（关键数据项、计算逻辑、时标顺序逻辑、权利方 / 义务方等）。两轨都用 Markdown 表格。
2. **业务模型图**（至少一张全景图 + 关键关系图）：
   - 终端轨 → **纯 ASCII 图**（画法见 [`./terminal-diagrams.md`](./terminal-diagrams.md)）
   - 报告轨 → **手写 SVG**（画法见 [`./html-report.md`](./html-report.md)，套用 [`./templates/report.html`](./templates/report.html)）
3. **端到端业务走查**：把模型放回真实流程，至少覆盖**正常流程 + 1 个异常 / 违约场景**，验证模型自洽。
4. **关键判定的依据**：为每个非显然的业务结论标注判定依据（如"标的物归参与者型，因属被凭证引用的 Party-Place-Thing""该违约设独立履约项，因违约方需主动退钱""平台是对手方，因直接收付抽佣"），让产出可追溯、可核查。**这是业务结论的支撑，不是方法的注脚**——禁"此处应用了 XX 理论 / 判别准则"式表述（详见上文「视角纪律」）。

## 报告结构层级：摘要全景 vs 详细局部

报告分两个清晰层级，避免把全局图和详细图混在同一粒度（多合同上下文系统尤其重要）：

- **摘要全景层**：系统的全局视图（合同上下文清单、最终业务模型全视图）——只露每个上下文的核心结构（参与方 / 核心履约项 / 跨上下文凭证串联），**不堆凭证字段、违约分叉等细节**。目的是一眼看全业务如何把领域系统转化为可盈利产品。
- **详细局部层**：单个合同上下文的完整建模（履约项表 + 凭证追溯图 + 违约流转图）——逐上下文一组，承载全部细节。

> 多合同上下文系统，详细层按「每个上下文一组卡片」组织（非按维度横向铺开），保持每个上下文的完整性与独立可读。8X Flow 的「总-分-总」三段式序列见 `bm-8x-flow` skill。

## 图的选型纪律：时序类必须画时序图（不得用列表替代）

**原则**：是否产出时序内容，由业务需要决定（不强制每份报告必有时序章节）；**但一旦产出时序类内容，形式必须是真正的时序图**——参与方纵向成列（生命线）+ 横向消息箭头（`-->` / `==>`）+ 时标约束 Note，**不得用有序列表 / 编号步骤替代**。

**什么是"时序类内容"**：在描述「多个参与方之间，随时间先后，谁向谁发出什么 / 产生什么凭证，并带时标先后约束」时，即为时序类。典型：端到端业务时序。

**为什么不能用列表**：时序的本质是参与方之间随时间推进的消息流。有序列表只表达"先 A 后 B"的线性顺序，丢掉时序建模要呈现的三样东西——**谁对谁**（生命线让参与方 / 角色显形）、**多方并行的等待与触发**（列表表达不了）、**时标约束贴在哪条消息上**（如 `paid_at > placed_at` 贴在消息上才不漏不错乱）。列表省的是画图功夫，丢的是时序语义；用时序列表替代时序图，等于这段内容白写。

**图 / 表 / 列表 的边界**：

| 内容性质 | 形式 | 例 |
| --- | --- | --- |
| 时序类（参与方消息流 + 时标） | **必须时序图**（报告轨 SVG / 终端轨 ASCII） | 端到端业务时序 |
| 结构性关系 | 画图 | 凭证链、合同对手方、违约分叉 |
| 字段清单 / 分类 | 表格 | 凭证表、四原型分类表、主要履约项表 |
| 验证叙事 | 列表 / 叙事可，但**不得冒充时序** | 端到端走查 |

> **端到端走查**（正常穿行 + 异常追溯）本身不是时序结构，**不强制**时序图；但其内容若实质在描述参与方交互，应升级为时序图或复用既有的时序图作锚点，不得另起列表把交互重新"步骤化"。

> 下文"关系密集、连线会乱时优先用表格"针对的是**字段清单 / 分类**（凭证表、四原型表），不适用于时序——时序不得以"关系密集"为由退化为表格或列表。

报告轨时序 SVG 画法见 [`./html-report.md`](./html-report.md) §六；终端轨 ASCII 时序图见 [`./terminal-diagrams.md`](./terminal-diagrams.md) §五.2；两轨语义一致。

## 终端轨 · ASCII 图原则（详见 terminal-diagrams.md）

- **节点框**：`+--+` 与 `|  |` 围成；标题 + 关键数据项分两三行。
- **线型语义**：`-->` 实线 = 主流程 / 数值来源；`..>` 虚线 = 规则来源 / 引用；`==>` 粗线 = 核心对手方结算。
- **对齐**：中文按 **2 个英文字符宽**计算 padding，保证框线对齐。
- **时序图**：参与方纵向成列，`->>` 实线消息、`-->>` 虚线返回，`Note:` 标注时标约束。
- **信息密度**：把关键约束（时标顺序、计算逻辑、权责方）标在线或节点上，图本身承载业务含义，不只是框图。
- **何时用表格替代**：关系密集、连线会乱时（如四原型分类、凭证表），优先用表格，不要硬画框图。

## 报告轨 · HTML + SVG 原则（详见 html-report.md）

- **结构**：Hero（标题 / 方法 / 场景）+ 侧栏目录 + 卡片化章节 + 端到端走查 + 待确认假设。直接套 [`./templates/report.html`](./templates/report.html)。
- **四原型配色**（贯穿全篇的视觉锚点）：时标型 = 粉 `#ec4899`、描述型 = 蓝 `#3b82f6`、角色型 = 黄 `#f9a825`、参与者型 = 绿 `#2e7d32`。
- **手写 SVG 图**：节点 = 圆角白卡 + 顶部色帽（原型色）+ 标题 / 关键数据项；连线 = 贝塞尔 + 箭头 marker；实线 / 虚线同终端轨语义。零 JS 依赖、矢量、离线可用。
- **不用 Mermaid**：报告里的图也用手写 SVG，保证可控、美观、离线。

## 实时预览（live server，默认模式）

产出过程中启动 live server 让 HTML 报告随建模推进**实时刷新**（浏览器边设计边显示）。完整机制（启动 / 增量更新 / 停止 / 异常处理）见 [`./live-session.md`](./live-session.md)。

- just-in-time 启动（需要看图时才提议），不默认开。
- **详细内容只写 HTML**（图、表、判别），**终端不重复完整方案**——每阶段终端只给一两句摘要（做了什么 + 关键判别 + 要确认的问题），省 token。
- 产出物：`报告.html`（完整方案）为主；终端 `.md`（ASCII 图）仅在**无 server**或用户要求时才产出，作兜底。

## 样例

- **看完整效果**：用 live server 实时生成（见 [`./live-session.md`](./live-session.md)），针对当前业务产出完整方案。
- **章节结构是默认推荐、可调**：按业务或用户需求增删、重排 section（标号由 CSS counter 自动重排）。
