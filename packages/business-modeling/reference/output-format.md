# 建模产出规范

所有业务建模 skill 的产出都应遵循本规范，保证产出**可视化、可核查、结构一致**。"业务建模至少要有一张能看的图"——纯表格或纯文字不算合格交付。

## 必含内容

1. **结构化表格**：凭证 / 合同 / 履约等关键字段用表格列清（关键数据项、计算逻辑、时标顺序逻辑、权利方 / 义务方等）。
2. **Mermaid 业务模型图**（至少一张全景图 + 关键关系图）：把模型可视化，让人一眼看懂业务结构。不能省略。
3. **端到端业务走查**：把模型放回真实流程，至少覆盖**正常流程 + 1 个异常 / 违约场景**，验证模型自洽。
4. **关键判别应用的说明**：标出在哪些点上应用了哪条判断准则（如标的物归参与者型、独立 vs 自动履约项、平台枢纽方判定），让产出可追溯、可核查。

## Mermaid 图原则

- **选型**：用 `flowchart` 表达结构关系（凭证链、合同关系、违约流转、边界划分）；用 `sequenceDiagram` 表达端到端时序。
- **标签**：节点 / 边标签含特殊字符（括号、`×`、`−`、`·`、`→`、`>` 等）一律用引号包裹，如 `id["标签(含括号)"]`、`A -->|"时标：x > y"| B`。
- **信息密度**：把关键约束（时标顺序、计算逻辑、权责方）标在边或节点上，图本身要承载业务含义，不只是框图。
- **线型约定**：实线 `-->` 表示主流程 / 数值来源；虚线 `-.->` 表示规则来源或引用；粗线 `==>` / `===` 强调核心（如平台对手方结算合同）。
- **分组**：用 `subgraph` 划分边界（如业务系统 vs 领域系统、不同履约上下文）。
