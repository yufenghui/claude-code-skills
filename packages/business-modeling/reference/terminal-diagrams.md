# 终端图表画法指南（纯 ASCII）

终端轨的图用**纯 ASCII** 表达，**不用 Mermaid**（终端里 Mermaid 代码块不渲染，是死代码）。本指南给出字符约定、对齐规则与各类标准图的范例。目标是：**任何终端 / IDE / issue 里直接可读，复制粘贴不丢对齐**。

## 一、字符约定

| 用途 | 字符 | 说明 |
| --- | --- | --- |
| 框角 / 交叉 | `+` | 节点框四角、连线拐角 |
| 横线 / 竖线 | `-` `\|` | 框边、连线 |
| 箭头 | `>` `<` `v` `^` | 流向 |
| 实线（数值 / 时标来源、主流程） | `-->` `->>` | 凭证链主链、时序消息 |
| 虚线（规则来源 / 引用） | `..>` `..` | 描述型规则 → 被引用的凭证 |
| 粗线（核心对手方结算） | `==>` `===` | 平台直接收付的结算合同 |
| 时序返回 | `-->>` | 异步 / 返回消息 |
| 标注 | `Note:` | 时标约束等注释 |

> 与报告轨 SVG 的语义**完全对应**：实线 ↔ SVG 实线、虚线 ↔ SVG 虚线、粗线 ↔ SVG 粗线。两轨只是渲染介质不同。

## 二、对齐规则

1. **中文按 2 个英文字符宽**计算 padding（等宽终端里一个汉字占 2 列）。
2. **框宽统一**：同一张图里同类型的节点框宽度一致，便于纵向对齐。
3. **宁可宽、不要挤**：框内左右各留 1 空格；长字段（如 `total_amount`）单独成行。
4. **零对齐压力的画法优先**：横向主链用「紧凑链式」（见下），避免横向多框紧贴的手工对齐。

## 三、两种节点表达

### A. 紧凑链式 `[ ]`（主链全景首选）

单行节点、行内箭头，对齐零压力，最适合横向凭证链 / 流程主线：

```
[Order 订单] --> [Payment 支付] --> [Shipment 发货] --> [Delivery 收货] --> [Settlement 结算]
 placed_at       paid_at           shipped_at          received_at        扣佣·担保放款
 total_amount    amount
```

- 第一行：`[中文名 英文标识] --> [...]`。
- 第二行起：每个节点正下方对齐写**关键数据项**（缩进到节点名下方）。
- 分叉 / 旁支：从节点下方用 `|` `+-->` 引出。

### B. 展开框 `+--+`（突出单个凭证细节）

需要列多个关键数据项 / 计算逻辑时，用框式纵向展开（一个框聚焦一个凭证）：

```
+------------------------+
| Order 订单   [时标型]  |
+------------------------+
| placed_at              |
| total_amount           |
|   = Σ(SKU价 × qty)     |
|     − Coupon − Promo   |
+------------------------+
```

- 顶行：中文名 + 英文标识 + `[原型]` 标签（时标型 / 描述型…）。
- 体内：关键数据项 + 计算逻辑（可缩进表达公式）。
- 描述型规则框同款，标签写 `[描述型]`。

## 四、线型语义（务必一致）

```
-->   实线：数值 / 时标来源、主流程        例 Order --> Payment
..>   虚线：规则来源 / 引用（描述型→凭证）  例 优惠券 ..> Order
==>   粗线：核心对手方结算合同（平台盈利）  例 商家 ==> 平台
```

## 五、标准图范例

### 1. 凭证链图（四色法 · 业务脊梁）

主链用紧凑链式，规则来源单独列一节用虚线：

```
凭证链（--> 数值/时标来源）

  [Order 订单] --> [Payment 支付] --> [Shipment 发货] --> [Delivery 收货] --> [Settlement 结算]
   placed_at       paid_at           shipped_at          received_at        扣佣·担保放款
   total_amount    amount
       |
       +--> [LogisticsFee 物流费]   fee = f(重量, 地区)

规则来源（..> 引用）

  [SKU 商品目录]  ..> [Order]
  [优惠券 Coupon] ..> [Order]
  [FreightRule]   ..> [LogisticsFee]
  [CommissionRule]..> [Settlement]
  [ReturnPolicy]  ..> [Delivery]
```

时标约束单独列（比塞进图里更清晰）：

```
时标顺序：
  paid_at     > placed_at        （先下单后支付）
  shipped_at  > paid_at          （先付后发；COD 是变化点）
  received_at > shipped_at
  settled_at  > received_at      （担保交易：收货确认触发放款）
```

### 2. 时序图（端到端）

参与方纵向成列，`-->` 实线消息、`==>` 粗线结算、`Note:` 标注约束：

```
用户            平台             商家            承运商
 |               |                |                |
 |--下单-------->|                |                |
 |--发起支付---->|                |                |
 |               |--支付确认----->| (自身)         |
 |               |--派单--------->|                |
 |               |                |--备货--------->|
 |               |                |--发货--------->|
 |               |                |                |
 |               |                |       ...送达/收货确认
 |<==============货款结算(扣佣)===================|
 |               |===========物流费结算==========>|

Note: paid_at > placed_at；settled_at > received_at（担保放款）
```

### 3. 合同图（8X Flow · 平台枢纽方）

`==>` 双向粗线 = 平台直接收付的结算合同（盈利来源）；普通线 = 平台不直接收付：

```
   订单交易（平台仅撮合）
 用户 <--------------------> 商家

 用户 <---- 支付协议 ----> 支付方

 商家      <== 货款结算(抽佣) ==>     平台(枢纽·对手方)
 承运商    <== 物流费结算     ==>     平台(枢纽·对手方)
```

### 4. 违约流转图（8X Flow）

从主要履约项分叉，标「独立履约项 vs 自动作废」，终止于打官司：

```
[主要履约项：支付 / 备货交付]
   |
   |-- 超时未支付 ----------> (自动作废) 订单关闭 --> [终止]
   |-- 已付款未发货取消 ----> [Refund 退款]          (独立履约项)
   |-- 已收货·七天无理由 --> [Return + Refund]      (受 return_window 约束)
   |-- 发货超时(KPI偏差) --> [TimeoutPenalty 罚扣]
   |                              |
   |                              +-- 拒不赔付 --> [打官司 --> 终止]
```

### 5. KPI 链图（成本结构入口）

目标凭证 ..对比.. 实际履约凭证 → 偏差触发违约凭证：

```
[ShipPromise 目标 deadline]  ..对比..  [ShipmentConfirm 实际 shipped_at]
                                                |
                                      shipped_at > deadline ?
                                          |-- 是 --> [TimeoutPenalty 罚扣]

[DeliveryPromise 目标 deadline] ..对比.. [DeliveryConfirm 实际 received_at]
                                                |
                                      received_at > deadline ?
                                          |-- 是 --> [TimeoutPenalty 罚扣]
```

### 6. 业务模型全景图（8X Flow · 骨架全景，终端版）

对应报告第三段骨架全景的终端版：每个合同上下文画**履约请求→确认 + 权责方 + 凭证**，标的物归领域、变化点（角色化履约确认）、四类边界。按上下文数量选布局（单上下文→纵向、≥2→网格）。**凭证字段 / 追溯链 / 违约分叉仍在逐上下文详细图里**，全景不重复：

```
   [创作协议]                  [读者订阅]                 [CMS 领域系统]
    极客时间↔作者              读者↔极客时间              （标的物：专栏/文章）
    交稿: 交稿请求──>交稿确认    支付: 支付请求──>支付确认◆    │
    分成: 分成单──>分成确认     访问: 开通请求──>访问确认      │ 标的物引用
             │                           │                       │
             │  分成依赖订阅付款           │ 支付确认角色化(变化点①) │
             +<──凭证(付款确认)───────────+                       │
                                         │                       │
                                         v                       v
                          [支付协议 / 预充值]
                           读者↔支付供应商   ──扮演支付确认──>+

   --> 实线 = 履约请求→确认 / 凭证跨上下文引用；==> 粗线 = 对手方结算；..> 虚线 = 标的物引用（领域系统）
        ◆ = 角色化履约确认（变化点，由其他合同凭证扮演）
```

要点：每个上下文框内画履约请求→确认（非只列履约项名）；框间用凭证引用线串联（体现跨上下文协作与变化点①角色化履约确认）；右接领域系统（标的物经凭证引用参与业务）。与报告轨 SVG 骨架全景语义一致（画法见 `html-report.md`「业务模型全景图画法」）。

### 7. 服务边界溯源图（8X Flow · 微服务划分依据）

§9 微服务设计的**首要图**（对应报告轨同名图），回答"服务边界从哪来"。左三类边界来源 → 右服务，实线映射边界来源；**不画服务间链接**（本节聚焦划分依据，非服务生态拓扑）。与服务划分表互补（图=映射关系、表=字段清单）：

```
服务边界溯源（--> 边界来源映射）          真伪：◉ 真 / ◎ 伪

  [合同] 订单合同   ──┐
  [合同] 支付合同   ──┤
  [履约] 支付履约   ──┼──▶  ◉ 订单服务    /orders
  [履约] 配送履约   ──┤    ◉ 支付服务    /payments
  [领域] 商品目录   ──┤    ◎ 配送服务    /delivery (分布式服务)
  [领域] 运力       ──┘

  --> 实线 = 边界来源映射（按业务能力聚合）；不画服务间超媒体链接
```

要点：左列按 `合同 / 履约 / 领域` 三类分组（来自 §3），实线汇聚到右侧服务体现"按业务能力划分"；服务节点标真伪（呼应三关键点 + 四种伪模式）。与报告轨 SVG 语义一致（画法见 `html-report.md`「微服务边界溯源图画法」）。

## 六、何时用表格替代图

关系密集、画框会乱的，**优先用 Markdown 表格**，不要硬画：

- **凭证表**（关键数据项 + 计算逻辑 + 时标顺序逻辑）→ 表格。
- **四原型分类表**（元素归时标型 / 描述型 / 角色型 / 参与者型）→ 表格。
- **主要履约项表**（权利方 / 义务方 / 请求段 / 确认点）→ 表格。
- **合同拆分表**（甲方 / 乙方 / 标的物）→ 表格。

图只画「结构性关系」（凭证链走向、合同对手方、违约分叉、时序）；「字段清单」用表。

## 七、检查清单

- [ ] 没有 Mermaid 代码块。
- [ ] 线型语义统一（`-->` / `..>` / `==>`）。
- [ ] 关键约束（时标顺序、计算逻辑）要么在图上、要么紧跟图后列出。
- [ ] 关系密集处用了表格而非硬画框。
- [ ] 终端等宽字体下可读、框线基本对齐。
