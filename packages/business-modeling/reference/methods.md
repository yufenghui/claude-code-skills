# 建模方法地图

本课程提供多种建模方法，各有适用场景。**先选对方法，再执行步骤**。本文件给每个方法的"是什么/适用/核心步骤提要/出处"；详细引导步骤见对应 skill。

## 方法选择指南（按你要解决的问题）

| 你的场景 | 首选方法 | 备注 |
| --- | --- | --- |
| 业务系统，要建模**盈利流程**（收入/成本/权责） | **8X Flow** | 业务逻辑建模王牌；从合同履约出发 |
| 业务系统，要找**关键领域事件/凭证链**（强分析） | **四色建模法** | 从收入流/成本结构推演业务脊梁 |
| 业务系统，想**与业务方共创、快速发散** | 事件风暴（Event Storming） | 弱分析，依赖主持人经验，初学者易废 |
| 要让模型**变成业务方读得懂的统一语言** | 角色-目标-实体法 / 催化剂法 | 共创流程 |
| 不确定模型与实现是否健康 | DDD 健康度评审（两关联一循环体检） | 评审型，非建模 |
| 想把已有模型**下沉为中台可复用模式** | 中台宏流程提取（**8X Flow 延伸**） | 非独立方法；见 `bm-8x-flow` 延伸章节 |
| 要给云化服务**设计 SaaS 选项分档** | 魔球建模法 | 高价值/现金奶牛/尴尬三类 |

**关键判别**：先区分是**领域系统**（运营无关，如搜索引擎/CMS）还是**业务系统**（与盈利相关）。领域系统用对象建模/知识消化即可；业务系统优先 8X Flow，四色法作为发现凭证的子方法嵌入其中。

## 1. 知识消化（DDD 总纲，非独立 skill）
- **是什么**：Eric Evans 的"两关联一循环"，是 DDD 的核心过程，也是其他建模方法的前提。
- **适用**：作为所有 DDD 建模的底层框架；单独用时适合领域系统。
- **核心**：模型↔实现关联、统一语言↔模型关联、提炼知识循环。
- **出处**：01、02、03。

## 2. 催化剂建模法（Catalyst Modeling）
- **是什么**：把交互（流程/用例）显式建模进对象图，用 uses 关系给对象打标签，展开流程维度。
- **适用**：让模型反映业务流程维度；但因"交互无对应实现组件"，在当代架构中略尴尬。
- **出处**：07。

## 3. 角色-目标-实体法（Role-Goal-Entity）
- **是什么**：与业务方共创模型的流程。
- **步骤提要**：① 梳理所有角色 → ② 澄清各角色目标 → ③ 提取目标中的名词/实体 → ④ 围绕实体建模 → ⑤ 向业务方解释模型如何实现目标（模型即统一语言）。
- **适用**：DDD 共创场景，比催化剂法更实用。
- **出处**：07。

## 4. 事件建模法 / 事件风暴（Event Storming）
- **是什么**：用事件表示行为、按时间线组织，与业务方共创。
- **两条原则**：① 用事件表示行为（事件成对出现暗示行为）；② 用多条时间线划分粒度（**交互事件** vs **领域事件**，领域事件=引起领域对象变更/聚合根行为接口）。
- **事件风暴流程**：头脑风暴找领域事件 → 找命令与行动者（AIC）→ 找策略与 SIC → 找聚合与阅读模型 → 细化领域模型。
- **命名规约**：事件"名词+动词被动式"（Order Paid），命令"动词+名词"（Pay Order）。
- **局限**：弱分析，收敛逻辑语焉不详，"一学就会一用就废"。
- **出处**：08。

## 5. ⭐ 四色建模法（4 Color Modeling）—— 独立 skill
- **是什么**：徐昊自创的**强分析法**，从收入流与成本结构的业务底层逻辑出发，推演凭证链（业务脊梁），产出四原型模型。
- **适用**：业务系统，需要可推演、可验证、易被业务方接受为统一语言的模型。比事件风暴更确定、更不依赖主持人经验。
- **三条核心逻辑**：现金收入承义务 / 现金支出有权利 / 无现金则目标-实际对比（KPI）。
- **步骤提要**（详见 `bm-four-color` skill）：
  1. **找现金往来/KPI**，构造凭证，列关键数据项（时间点 + 金额）。
  2. **追溯前序凭证**：对每个关键数据项问"它从哪来"，找到提供它的前序凭证，直到人工输入；记录凭证间的**计算逻辑**与**时标顺序逻辑**。
  3. **找履约凭证**：现金权责对应的权利/义务，引入表示权利的凭证（如订阅），约束其时序（如 started_at > 支付完成）。
  4. 得到**业务脊梁**（凭证链 = 领域事件流）。
  5. **验证**：凭证转单据 + 角色扮演游戏模拟经营（重点跑异常追溯），看能否证明权责。
  6. **补全**：找角色 → 参与者 → 标的物 → 补描述对象，按关键数据项关联连成领域模型。
  7. **分类**：模型元素归入四原型（时标型/描述型/角色型/参与者型）。
- **两条硬约束**：① 凭证必须围绕现金往来或 KPI；② 凭证间必须通过关键数据项明确关联。
- **出处**：09（追溯法亦见 08）。

## 6. ⭐ 8X Flow（合同履约法）—— 独立 skill
- **是什么**：徐昊自创，用"合同（最小业务上下文）+ 权责履约（最小业务交互）"建模业务系统，天然识别变化点与弹性边界。
- **适用**：业务系统（与盈利相关），尤其业务平台/中台；模型易映射到微服务。
- **元模型**：合同上下文（两个角色）/ 履约请求（时间段）/ 履约确认（时间点）/ 凭证 / 标的物（来自领域系统）。
- **五步法**（详见 `bm-8x-flow` skill）：
  1. **找合同上下文**，明确参与方（合同只在两方之间；多方=多份双方合同）。
  2. **找主要履约项**，按四色建模找凭证（履约请求=时间段，履约确认=时间点）。
  3. **找违约情况**，设新履约项，按四色找凭证。
  4. **重复第 3 步，直到不得不打官司为止**（终止条件）。
  5. **把参与方与标的物划入领域边界**。
- **引入变化点**：核心合同依赖支撑合同是坏味道 → **让履约确认角色化**（能力供应商/反转依赖），让其他合同凭证扮演它。
- **弹性边界落点**：履约上下文 = 弹性边界（合同上下文不是）；领域上下文也是；渠道上下文独立成界（**四类天然边界**：合同 / 履约 / 领域 / 渠道上下文，见 `concepts.md` §五）。
- **出处**：14（领域/业务分离）、15（五步法+变化点）、16（渠道上下文+全生命周期+无合同场景）。

## 7. 中台宏流程提取（8X Flow 的延伸，非独立方法）

> **定位**：中台建模用的是 8X Flow 本身（17 / 18 讲开篇原话："如何使用 8X Flow 建模中台系统"），故本方法是 **8X Flow 的下游延伸**，不是与之并列的独立方法。完整引导见 `bm-8x-flow` skill 的「中台化：宏流程提取」延伸章节。
> **前置**：① 先理解中台概念（中台 / 前台 = 团队 / 效能-自主 / 宏流程，见 `concepts.md` §六）；② 已有一份完整的 8X Flow 业务模型——宏流程从**已存在模型**提取，不直接建模。

- **适用**：已有 8X Flow 业务模型，且目标是把可复用业务模式下沉为中台（支撑多个前台团队）。
- **步骤**（18 讲）：① 8X Flow 建模 → ② 确定要复用的业务模式 → ③ 找对应合同上下文 → ④ **将履约确认项角色化**引入变化点 → ⑤ 在不同场景扩展 / 替换变化点 → ⑥ 各上下文继续 8X Flow 建模直到能解释清楚。
- **关键约束**：可对**履约确认项**角色化，**不能对履约请求项**角色化（破坏权责语义）；防过度抽象——从已存在模型提取，而非凭空建模。
- **出处**：17（中台定义、宏流程）、18（提取步骤、角色化约束）。

## 8. 魔球 SaaS 建模法（Money Ball Offering Modeling）
- **是什么**：围绕"选项=能力+服务质量"做 SaaS 化，按三类客户设计选项分档。
- **三类客户**：高价值（重响应，独立组网）/ 现金奶牛（覆盖大多数，降成本）/ 尷尬（尽量转化为前两类）。
- **存量系统**：先建模成"遗留选项"放入尴尬档（100% 用户=尴尬=潜力无穷），再分化变革。
- **出处**：20。

## 共用的判断标准（所有 skill 通用）

- **是不是在做 DDD**：模型与软件实现是否关联（修改模型=修改代码）。
- **贫血 vs 充血**：业务逻辑富集于聚合根，还是散落于 DAO。
- **模型是否健康**：有没有上下文过载（User 巨类）、领域层是否偷偷依赖基础设施、API 是从数据角度（REST）还是行为角度（RPC）驱动。
- **耦合好坏**：必然耦合（业务必需，对隐藏业务能力）vs 偶然耦合（对具体实现）；抽象必须在业务上找到对应能力，否则是耦合 PTSD。
- **模式使用**：先确认"要解决的问题"再配解决方案；不能从代码结构反推模式。
